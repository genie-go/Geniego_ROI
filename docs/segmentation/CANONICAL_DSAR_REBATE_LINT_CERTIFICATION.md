## §3. 엔티티 정의

### E-01. `LintCertificationRun`

`run_id` · **`commit_sha`** · `measured_at` · `criteria_version` · `verdict` · `evidence`.

**`commit_sha` 결속 필수**(5-8 D-9 동일 정본) — **코드가 바뀌면 인증은 무효**다.
**"작년에 인증받았다"는 증거가 아니다.**

### E-02. 🔴 `CertificationReadiness` — 발급 전 게이트

```
readiness = NOT_READY | PARTIAL | READY
규칙: readiness != READY 이면 인증서 발급 금지 (verdict 생성 자체를 막는다)
```

**"조건부 인증"·"임시 인증"을 두지 않는다.** 그런 등급은 **반드시 "인증됨"으로 인용**된다.

> 데이터 헌법 V3의 **`READY`/`WARNING`/`BLOCKED`** 3단계와 동일 철학이다 —
> **품질 미달 데이터를 AI가 쓰지 못하게 하듯, 미달 Lint 체계로 인증서를 내지 않는다.**

### E-03. `CertificationScope` — 무엇을 인증했는지 명시

**부분 인증을 숨기지 않는다.** 1-7이 인증할 수 있는 것은 **§53 규칙을 영속한 2개 블록뿐**이고,
**나머지 7개 블록은 인증 범위 밖**이다. → `certified_blocks` / **`uncertifiable_blocks`** 병기.

**`uncertifiable_blocks`를 명시 필드로 두는 이유:** 안 적으면 **"1-7 인증 통과"가
06-A 전체를 덮는 것으로 읽힌다.** 실제로는 2/9다.

## §4. 1-7이 인증하지 못하는 것 (정직 표기)

1. **규칙의 타당성** — 81개가 **옳은** 규칙인지는 인증 대상이 아니다. **틀린 규칙도 100% 배선될 수 있다.**
2. **7개 블록** — Lint를 영속하지 않았으므로 **인증 범위 밖**(§3 E-03).
3. **미작성 규칙** — 있어야 할 규칙이 없는 것은 **Lint가 스스로 못 잡는다**.
   1-1/1-2/1-3에 Lint가 0인 것을 **Lint로 발견할 수 없다** — 사람이 세야 했고, **1-7이 세었다**.
4. **1-7 자신** — 1-7 스펙 미수령(자율 판단) → **1-7의 요구 목록도 저장소에 없다**(1-6 COV-GAP-01 적용).
