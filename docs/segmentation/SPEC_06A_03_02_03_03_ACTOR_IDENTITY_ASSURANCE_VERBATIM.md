## §0 작업 목적

Decision Integrity Core·Immutable Ledger·Cryptographic Hash Chain·Tamper Detection 위에 **Actor Identity Assurance & Authentication Binding Governance**를 구축한다. 핵심 목적: Approval Decision을 수행한 Actor가 실제 누구인지, 어떤 인증수단·세션으로 제출했는지, 그 인증상태가 Commit 시점까지 유효했는지를 **Canonical Identity + Immutable Evidence로 고정**. 단순 user ID 저장을 넘어 다음에 답한다: Actor의 Canonical Subject·로그인 Principal↔Decision Actor 동일성·사람/서비스/시스템 구분·Decision 시점 Active/Employment/Role/Tenant/Legal Entity 유효성·직접승인 vs Delegate/Impersonation·인증방식(password/SSO/Passkey/MFA/cert)·AAL 충분성·세션↔Command 결합·타세션 Token 재사용·인증후 Actor 변경·세션 탈취/재생·Device 신뢰·API Client 등록·Service Account 사람가장·System Actor 정책허용·인증 Context 불변저장·인증 Evidence 변조·만료후 Commit·Commit 직전 재검증·비활성/퇴직자 Commit·Privilege Escalation 직후·Cross-Tenant/Legal-Entity·과거 Decision을 현재 Identity로 재해석 없이 재현.

## §5 핵심 원칙

- **§5.1** Principal(인증된 주체) ≠ Actor(실제 수행 주체). Delegation/Administrative Acting/Support/Impersonation/Service-to-Service/System/Batch/External Integration 시 상이 가능. **Original Principal + Effective Actor 모두 보존.**
- **§5.2** Account ≠ Person. Account·External Identity·Canonical Subject·Person·Employment·Tenant Membership·Role·Position·Organization·Legal Entity 명시 연결. **Email만으로 Person 식별 금지.**
- **§5.3** 인증 성공 ≠ 승인 권한. Authentication="누구인가", Authorization/Approval Authority="무엇을 할 수 있는가".
- **§5.4** Actor Identity는 **Commit 시 재검증**(Account/Subject/Employment/Tenant/Legal Entity/Org Membership/Role/Position/Session/AAL/Device/Token/Impersonation/Delegation Status).
- **§5.5** Session↔Decision Command 결합(타세션 Token/Command 재사용 차단).
- **§5.6** Authentication Context=Immutable Snapshot 보존(과거 Decision에 현재 세션 재조회·재해석 금지).
- **§5.7** Raw Credential 저장 금지: Password·OTP원문·TOTP Secret·Refresh Token·Access Token 전체·Private Key·Client Secret·API Key 원문·Session Cookie·WebAuthn Private·Cert Private Key·Biometric. → Reference·Masked·Digest·Verification Result만.
- **§5.8** Impersonation 숨기지 않음(Original Principal 지우고 Effective Actor만 저장 금지).
- **§5.9** Service Account/System Actor는 사람처럼 승인 불가(명시 System Decision Policy + 제한 Action Scope 필요).
- **§5.10** Assurance Level ↔ Action Risk 연결(고액/결제/정산/계약/법률/보안/관리자취소/Correction/Supersession에 높은 AAL 요구 가능).
- **§5.11** Client 전달 Identity Claim(Header/Body/Query Actor ID)을 인증 Context 없이 신뢰 금지.
- **§5.12** 고객설정 비활성 불가: Canonical Subject Resolution·Original Principal/Effective Actor Preservation·Tenant Binding·Session Binding·Commit-time Revalidation·Disabled/Terminated Blocking·Raw Credential Storage Prohibition·Impersonation Disclosure·Identity Snapshot·Authentication Evidence·Identity Audit.
