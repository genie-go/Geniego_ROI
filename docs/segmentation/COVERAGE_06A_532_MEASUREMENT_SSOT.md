## §1. 이 문서가 측정기를 가리키는 이유

289차 ②가 확립한 규율의 직접 적용이다: **"값이 틀렸다"가 아니라 "값이 문서에 손으로 박혀 있었다"가 근본**이다. `351`은 **정정값 `455`조차 이미 `448`로 어긋난 채** 문서 15편에 복제돼 있었다.

> **커버리지는 특히 위험하다** — 분모(원문 항목)와 분자(판정)가 **둘 다 문서 안에** 있어서, 손으로 적는 순간 **자기참조로 굳는다**. 산출 문서가 바뀌어도 요약 숫자는 안 바뀐다. ∴ **숫자는 이 저장소 어디에도 박지 않는다.**

---

## §2. 분모의 정의 (역산 금지)

**분모 = 스펙 원문 항목명.** [`SPEC_06A_4_5_3_1_5_3_2_VERBATIM.md`](SPEC_06A_4_5_3_1_5_3_2_VERBATIM.md) §75가 나열한 84편의 **`## 1. 원문 전사 + 판정` 절에 있는 번호 표 행**만 계수한다.

**계수 대상이 아닌 것**(측정기가 절 경계로 배제):
- `## 0. 현행 실측` 표 — **현행 코드 인벤토리**이지 요구가 아니다
- `## 2. 규칙` — 도출된 규율
- `## ★정정 …` 절 — 9회차 통지 판정 정정(D-11)

🔴 **역산 금지**(289차 ⑤): 산출물이나 현행 코드에서 요구를 역산하면 **분모=분자 동어반복 → 커버리지가 정의상 100%** = 가짜녹색의 가장 순수한 형태.
🔴 **개수는 분모가 아니다**(5-3-1 3단계 교훈): `REQUIREMENT_TYPE`은 **20/20 개수가 일치했는데 축 자체가 날조**였다. 분모 검증은 **개수가 아니라 항목명 원문 대조**다. 측정기는 **개수만** 세므로, **항목명 정합은 측정기가 보증하지 않는다** — 그것은 ⓔ 인용검증의 몫이다.

## §3. 커버 판정의 정의 (인접 자산 ≠ 요구 충족)

**분자 = `VALIDATED_LEGACY` 만.**

| 판정 | 커버? | 의미 |
|---|---|---|
| `VALIDATED_LEGACY` | ✅ **커버** | 현행이 요구를 **실제로 충족** → 확장(재구현 금지) |
| `LEGACY_ADAPTER` | ❌ | **인접 자산이 있을 뿐** 요구 미충족 → 위임 대상 |
| `KEEP_SEPARATE_WITH_REASON` | ❌ | **형태 유사·도메인 상이** → 매핑 금지 |
| `NOT_APPLICABLE` | ❌ | 현행 부재 → 신설 |
| `CONTRACT_ONLY` | ❌ | 계약만 확정 · **실 코드 0** |
| `VACUOUS` | ❌ | 코드는 있으나 **도달 불가**(생산자 0) |
| `BLOCKED_*` | ❌ | 선결 미이행 시 차단 |

> 🔴 **`LEGACY_ADAPTER`를 커버로 세면 갭이 정의상 소멸한다.** "인접 자산이 있다"와 "요구를 충족한다"는 다르다. 5-3-2에서 이 혼동의 대표 사례가 **`claim*` 5계열**(워커의 잡 선점 ≠ 사람의 Task 클레임 · 주체·대상·목적·인가·해제 5축 상이)과 **`journeys` 노드 13종**(마케팅 도메인)이다.

**병기 행 처리 — 보수적 채택.** 판정 셀에 `VALIDATED_LEGACY`와 비-커버가 **함께** 적힌 행(인접 자산은 실재하나 승인 도메인 요구는 미충족)은 **비-커버로 계수**한다. 커버 **과대계상 방지**가 목적이며, 산출 커버리지는 **하한(floor)** 이다.

## §4. 재현 · 이상 항목

```
node tools/measure_06a_532_coverage.mjs          # 요약
node tools/measure_06a_532_coverage.mjs --json   # 전수(파일별·이상항목 포함)
```

측정기가 스스로 보고하는 이상 항목(숫자를 조용히 맞추지 않기 위해 **감추지 않는다**):
- `MULTI_VERDICT` — 판정 병기 행. 보수적 채택되므로 **커버리지에 유리하게 작용하지 않는다**.
- `NO_VOCAB` — 정식 판정 어휘가 없는 행. **어휘 드리프트**(예: "부분 재사용")이며 **커버로 계수되지 않는다**. 발견 시 정식 어휘로 정정할 것.
- `NO_SECTION1` — `## 1. 원문 전사` 절이 없는 문서. **원문에 항목 축 자체가 없어 표를 비운 정직한 경우**가 있다(§35 `value type` 열거 없음 · §36 `status` enum 없음 · §55 필수 필드 축 없음). **이 경우 지어내지 않은 것이 정답이며, 분모 0이 결함이 아니다.**

## §5. 이 값을 읽는 법 (오독 금지)

1. **커버리지가 낮은 것은 결함이 아니다.** 5-3-2는 **승인 워크플로 엔진 자체가 부재**한 영역이다(워크플로 정의 테이블 `workflow_*`/`flow_*`/`wf_*` **grep 0**). 낮은 커버리지는 **정직한 부재 기록의 결과**다.
2. **높이려 하지 마라.** 커버리지를 올리는 유일한 정당한 방법은 **실 코드 구현**이다. 판정을 `LEGACY_ADAPTER`→`VALIDATED_LEGACY`로 재분류해 올리는 것은 **역산이며 가짜녹색**이다.
3. **§64 Critical Gap 26/26 `NOT_APPLICABLE`을 "갭 0 = 양호"로 읽지 마라.** **판정 대상 개념 자체가 부재**하다는 뜻이며, 신설 시 26종이 **즉시 High/Critical로 활성화**된다.
4. **5-3-2는 "구축 완료"가 아니라 "계약 명세 확정"이다.** 실 코드·테이블·노드 **0건** · §82 검증 게이트 통과 **0/34** · §65 Lint 28 / §66 Guard 37 **전건 `CONTRACT_ONLY`**.
5. **`ENFORCED(예방)` 등급은 현행 레포에 없다** — 브랜치 보호 + required check 미설정(G-06b) → 가드는 **탐지일 뿐 예방이 아니다**.

## §6. 측정기가 보증하지 않는 것 (정직 표기)

- **항목명 정합** — 측정기는 개수만 센다. 원문 항목명과의 대조는 **ⓔ 인용검증**의 몫이다(5-3-1 3단계 교훈: 개수 일치가 정합을 보장하지 않는다).
- **`file:line` 앵커의 유효성** — 코드가 바뀌면 stale 이 된다. 9회차에 **공통 규율 파일의 앵커 6건이 실측과 달랐다**(`Alerting::dispatch` 팬텀 · `JourneyBuilder:554`→`:551/:556` · `Mapping :209`→`:210` · `customer_id 필수`→nullable · `claimSendOnce :672`→정의 `:450-461`/호출 `:679` · `journey_node_logs` tenant 보유). **앵커는 인용 시점마다 재실증하라.**
- **판정의 타당성** — 측정기는 **적힌 판정을 셀 뿐** 그 판정이 옳은지 모른다.

> ★**측정기가 있다는 것이 값이 옳다는 뜻은 아니다.** 측정기는 **손으로 박은 값이 복제 수로 이기는 구조**만 없앤다.
