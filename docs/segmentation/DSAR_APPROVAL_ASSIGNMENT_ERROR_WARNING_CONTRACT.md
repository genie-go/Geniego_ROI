## 2. 기존 구현 대조

§GROUND_TRUTH 기준: Approval Assignment Engine 자체가 ABSENT 이므로 위 Error/Warning 계약을 방출하는 코드 경로는 부재하다. 인접 실존 자산의 실패 신호는 **표면적으로만** 겹친다:

- `catalog_writeback_job` 승인큐(`Catalog.php:75-84`) — 상태 `pending_approval→queued→processing→done/failed`(`Catalog.php:396,2383-2407`)로 실패를 표현하나, 위 정규화 Error 코드 체계(Authority/Delegation/SoD/Lease/Fencing)를 방출하지 않는다. CAS claim(`Catalog.php:1721-1731`)·600s 회수(`Catalog.php:1699-1702`)는 `APPROVAL_STALE_LOCK_DETECTED`류 조건에 인접하나 Fencing Token 부재.
- `omni_outbox`(`Omnichannel.php:95-99,405,425-448`) — CAS/`FOR UPDATE SKIP LOCKED` claim 존재하나 fencing 없음 → `APPROVAL_FENCING_TOKEN_STALE` 계약의 substrate로만 인접.
- `TeamPermissions DELEGATION_EXCEEDED`(`TeamPermissions.php:627-647`)는 RBAC 부여상한 monotonicity 오류로, 위 `APPROVAL_DELEGATION_*` 계약(위임 정의/기간/천장)과 **의미가 상이**하다(재사용 금지).
- SoD hook·CoI·Actor Snapshot foundation 부재(축4 PARTIAL) → `APPROVAL_SOD_VIOLATION`·`APPROVAL_CONFLICT_OF_INTEREST`·`APPROVAL_DECISION_ACTOR_MISMATCH`는 방출 근거 자체가 없다.

## 4. 확장/구현 방향 (설계)

- 순신규 Error/Warning enum 을 단일 정본 상수로 정의(문자열 하드코딩 금지). 방출 지점은 §60 Runtime Guards·§58 Critical Gap Policy 게이트에 1:1 대응시킨다.
- Mandatory Control: 모든 ERROR 는 차단(fail-closed)·모든 WARNING 은 우선순위/제외 신호로만 사용(권한 부여 금지, §36 원칙 준수).
- 무후퇴: `catalog_writeback_job`·`omni_outbox` 의 기존 실패표현(`failed`, claim 만료)은 신규 Error 코드로 **치환하지 말고** 매핑 레이어에서 병행 노출. 선행 4축 신설 전에는 축 의존 Error(Authority/Delegation/SoD)를 실제 방출하는 코드 배선 금지 — 스펙 상수만 예약.
