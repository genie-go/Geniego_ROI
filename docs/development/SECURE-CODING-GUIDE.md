# 보안 코딩 가이드

CCIS Part004 | 2026-07-22

이 문서의 규칙은 **가상의 위협이 아니라 이 저장소에서 실제로 발생한 사고**에서 나왔다.
각 항목에 근거 사례를 함께 적는다.

---

## 1. 멀티테넌트 격리 — 최우선

이 서비스의 단일 최대 리스크다. 데이터가 섞이면 되돌릴 수 없다.

**규칙**

- 테넌트는 **세션 토큰에서 해석한다.** `X-Tenant-Id` 헤더는 클라이언트가 조작할 수 있으므로 신뢰하지 않는다.
- 모든 조회·변경 SQL 에 `tenant_id` 조건을 넣는다. **조인 대상 테이블에도** 넣는다.
- 공용 데이터는 `__shared__`/`demo` 처럼 명시적 스코프로 분리하고, 읽기 경로에서도 그 스코프를 쓴다.
- 관리자 대행(act-as) 기능은 요청 시점의 전역 테넌트 해석을 덮어쓸 수 있다 —
  대행 컨텍스트가 다른 사용자의 요청에 새지 않는지 확인한다.

**근거 사례**

- 롤업 집계가 테넌트 격리 없이 공개 bypass 로 노출돼 있었다(204차 재구축).
- 공용 카탈로그를 실테넌트 스코프로 읽어 상품마다 3MB 를 재수집해 502 를 유발했다(285차).
- act-as 전역 테넌트 하이재킹으로 특정 role 의 전 메뉴가 공백이 된 사례가 있다.

---

## 2. 인증 · 세션

- 세션 조회는 **반드시 `hashToken(입력)`** 으로 한다. 저장된 해시와 raw 토큰을 직접 비교하지 않는다.
  raw 와 sha256 은 **둘 다 64-hex 라 형식만 보고는 구분할 수 없다** — 코드를 읽어 확인한다.
- 실패는 fail-secure 로 처리한다. 조회 실패·플랜 확인 실패 시 권한을 열어주지 않는다.
- OTP·인증 시도에는 스로틀을 건다.
- 브레이크글래스 계정 정보를 예제·문서·로그에 노출하지 않는다.

**근거 사례**: raw 토큰 비교로 세션 전면 401 + 저장 해시 replay 가 동시에 가능했던 P0 (289차후속).

---

## 3. 인가 (Authorization)

- 인증과 인가를 분리해 검사한다. 로그인했다는 사실이 접근 권한이 아니다.
- 리소스 소유권을 확인한다(IDOR).
- 쓰기 작업은 서버에서 role/scope 를 강제한다. **UI 에서 버튼을 숨기는 것은 보안이 아니다.**
- 승인이 필요한 작업은 **모든 경로**에서 승인 게이트를 통과하게 한다.

**근거 사례**: 직접 publish 경로 3개가 정책 평가를 우회해 고액 발송이 무승인으로 나갔다(289차).

---

## 4. 주입 (Injection)

- **DML 은 예외 없이 Prepared Statement.** 값 보간 금지.
- DDL 에서 컬럼명을 보간하는 기존 자가치유 패턴은 **보간 대상이 코드 내 리터럴 배열일 때만** 허용한다.
  요청에서 온 값은 식별자로도 절대 쓰지 않는다.
- 사용자 입력을 셸 명령·파일 경로로 넘기지 않는다.
- 정규식에 사용자 입력을 그대로 넣지 않는다(ReDoS).

---

## 5. 외부 요청 (SSRF)

- 사용자가 준 URL 로 서버가 요청하기 전에 스킴·호스트를 검증한다.
- 사설 대역·메타데이터 주소를 차단한다.
- 리다이렉트를 따라갈 때 재검증한다.

**근거 사례**: 알림 연동에서 SSRF 가 발견돼 수정됐다(279차).

---

## 6. 시크릿

- 소스·문서·로그·응답에 실제 시크릿을 넣지 않는다.
- 설정은 저장소 루트 `.env`(`GENIE_*`)에서 읽는다. `.env` 는 gitignored 다.
- 예제 파일에는 **플레이스홀더만** 넣는다. bootstrap 도 시크릿을 생성하지 않는다.
- 커밋 전 훅의 시크릿 스캔을 `--no-verify` 로 우회하지 않는다.
- 자격증명을 회전하면 운영·데모 **양쪽** `.env` 를 동시에 갱신한다.

---

## 7. 출력 · 오류 노출

외부 응답에 담지 않는다:

스택 트레이스 · SQL · 스키마 · 내부 호스트/경로 · 클래스명 · 토큰 · 외부 서비스 크리덴셜

- 오류 코드(`TENANT_ACCESS_DENIED`)와 사용자 메시지를 분리한다.
- 로그에 비밀번호·토큰·PII 를 남기지 않는다.

---

## 8. 프론트엔드

- `target="_blank"` 에는 `rel="noreferrer"` (리버스 탭내빙).
- `dangerouslySetInnerHTML` 을 새로 도입하지 않는다. 불가피하면 sanitize 를 통과시킨다.
- 클라이언트 번들에 서버 시크릿을 넣지 않는다. `VITE_*` 는 **전부 공개**된다.
- 토큰을 URL 쿼리에 싣지 않는다(로그·리퍼러 유출).

---

## 9. 무결성 · 감사

- 감사 로그의 정본은 하나다. 별도 해시체인을 새로 만들지 않는다.
- **해시체인은 탐지일 뿐 방지가 아니다.** DB 관리자는 여전히 UPDATE 할 수 있다 —
  이를 "변조 불가"라고 문서에 적지 않는다.
- 승인·집행 이력은 실제로 기록되는지 확인한다. 생산자가 없는 소비자는 가짜 기능이다.

**근거 사례**: 집행 액션이 실제로는 아무것도 하지 않는데 성공으로 보고되던 사례(287차).

---

## 10. 의존성

- 잠금 파일을 커밋한다. 버전을 고정한다.
- 사용하지 않는 의존성을 제거한다.
- `security-scan.yml` 의 `npm audit` 결과를 무시하지 않는다.

---

## 11. 검증 명령

```bash
make quality                    # 구문·게이트
git grep -n "dangerouslySetInnerHTML" -- frontend/src
git grep -nE "target=\"_blank\"" -- frontend/src   # rel 누락 확인
git grep -n "tenant_id" -- backend/src | wc -l     # 스코프 적용 규모 확인
```

보안 관련 변경은 **반드시 코드를 읽어** 확인한다. 화면과 로그는 증거가 아니다.
