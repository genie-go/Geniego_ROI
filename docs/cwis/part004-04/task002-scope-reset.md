DDD 스트라텀 신설 = 저장소에 없던 **두 번째 아키텍처 도입**. 헌법 Golden Rule(Replace 아닌 Extend)·중복 구현 금지 위반.

### ⑤ ★구조적으로 죽은 코드가 된다

본 명세는 Controller·API·Repository 구현을 **금지**한다. 따라서 생성될 도메인 클래스 약 15개는 **호출부가 존재할 수 없다**.

→ 재발 방지는 메모리 `reference_reachability_check_dead_feature` 에 기록됨.

지어내지 않고 **코드 실측으로 확인된 항목만** 싣는다.

### BACKLOG-1 · 즐겨찾기 6개 이상은 볼 방법이 없다 (우선)

```
Sidebar.jsx:431   items.slice(0, 5)
"더 보기"·showAll·viewAll 경로   0건
```

즐겨찾기는 무제한 저장되지만 패널은 **5개만** 표시하고 나머지를 여는 경로가 없다.

★**이 조건은 현 차수 변경이 만들었다.** 이전에는 추가 진입점이 없어 아무도 즐겨찾기를 가질 수 없었으므로 제한이 드러날 수 없었다. 추가가 가능해진 지금부터 6번째 이후는 저장은 되는데 보이지 않는다.

- 현재는 최신순 정렬(`.reverse()`)로 **방금 추가한 항목은 항상 보이도록** 완화해 둔 상태
- 근본 해소는 (a) 더 보기 토글 (b) 표시 개수 상향 (c) 전체 목록 화면 중 택1
- 판단 근거가 되는 실사용 데이터가 없으므로 **임의 선택하지 않고 보류**

### BACKLOG-2 · 별표 버튼의 모바일 터치 타깃 미달

```
FavStar   fontSize 12~13px · padding 2px 5px  →  약 23×17px
.sidebar .nav-item   min-height: 44px !important   (행 자체는 준수)
```

행 높이는 44px를 지키지만 **클릭 가능 영역은 별표 버튼 박스뿐**이라 모바일 권장치(44×44)에 미달한다.

★**이것도 현 차수가 만든 항목이다.** 접근성 Gap(CAP-04 `aria-pressed`)을 메우면서 터치 타깃 항목을 새로 만들었다 — 정직하게 기록한다.

- 해소는 간단(패딩 확대 또는 히트박스 확장)하나 **행 높이 44px 유지**가 조건이다. 사이드바 아코디언 `maxHeight` 계산이 `items.length * _itemH` 라 행이 커지면 클리핑이 생긴다(무후퇴 조건).

### BACKLOG-3 · 드래그 순서 변경 (CAP-05)

ST10에서 LOW·P3. 실사용 요구 미확인이라 **의도적 미구현** 상태 유지. 표시 5개 제한 하에서 사용자 정렬의 실익이 불분명하므로 BACKLOG-1이 먼저 정리되어야 판단 가능하다.

## 5. Option A 실행 기록 (2026-07-24)

ST01 미구현 → ST02 BLOCKED → ST03 BLOCKED 3연속 후 사용자가 Option A(계열 종결 + PHASE_3 프론트엔드 잔여 처리)를 확정했다.

### 5.1 CAP-04 · aria-pressed — **이미 해소돼 있었음**

ST10 Gap 분석은 `aria-pressed` 0건으로 기록했으나, 그 실측은 커밋 `67dee1fe46a`(FavStar 신설) **이전** 시점이다. 현행 `Sidebar.jsx:33` 에 이미 존재하며 브라우저 실측으로 77개 별표 전부 올바른 상태(`★`=true / `☆`=false)를 노출함을 확인했다. **추가 작업 없음.**

### 5.2 BACKLOG-1 · 6개 이상 열람 경로 — **해소**

`items.slice(0, 5)` 를 제거하고 목록을 스크롤 컨테이너(`maxHeight: 240` + `overflowY: auto`)로 감쌌다.

"더보기" 버튼이나 별도 관리 화면을 **신설하지 않은** 이유:

1. 총 개수는 이미 탭 라벨(`⭐즐겨찾기 8`)과 헤더 배지에 노출돼 있어 "더 있다"는 신호가 존재한다
2. 스크롤은 신규 UI 개념·신규 i18n 키·임의 제품 결정이 필요 없는 최소 해법이다
3. 신규 메뉴 자제 원칙에 부합한다

즉 **실사용 데이터가 없어도 결정 가능한 해법**이었다 — 보류 사유였던 "방향 판단에 데이터 필요"는 더보기/관리화면 같은 제품 결정에만 해당한다.

`recents` 는 `useRecentVisits(_, 5)` 가 원천에서 5개로 제한하므로 영향 없음.

### 5.3 BACKLOG-2 · 모바일 터치 타깃 — **해소**

별표와 해제(✕) 버튼 모두 모바일에서 히트 영역 **44×44** 확보(WCAG 2.5.5). **글리프 크기는 그대로** 두고 히트 영역만 넓혀 시각 변화는 없다.

**행 높이 조건 충족** — 우려했던 아코디언 클리핑은 발생하지 않았다. `styles.css:797` 이 이미 `.sidebar .nav-item { min-height:44px !important }` 를 적용하고 있어 모바일 행이 애초에 44px 이상이었고, 실측 결과 변경 전후 모두 **정확히 44px**(77행 전부, 초과 0건)로 동일했다.

### 5.4 ★부수 발견 — 즐겨찾기 패널이 데스크톱에서 보이지 않았다

작업 중 실측으로 드러난 **선재(先在) 결함**이다.

```
패널 외곽 컨테이너 실측 높이 = 2px  (위아래 테두리 1px씩 = 내용 높이 0)
```

원인: `.sidebar` 는 `display:flex; flex-direction:column` 이고 패널은 그 flex 아이템인데, 패널에 `overflow:hidden` 이 있어 CSS 사양상 **automatic minimum size 가 0** 이 된다(`min-height:auto` → 0). 그래서 사이드바 세로 내용이 넘칠 때마다 `flex-shrink` 가 패널을 0 까지 눌러버렸다. 메뉴가 77행이므로 사실상 상시 발생.

**내 변경이 만든 것이 아님을 실측 대조로 확인했다** — 구 `slice(0,5)` 동작을 DOM 에서 재현했을 때도 동일하게 2px 였다(`flex-shrink:0` 을 주면 297px 로 정상 렌더).

`67dee1fe46a` 가 고친 "즐겨찾기 도달 불가" 와 **같은 계열**(구현은 있으나 사용자에게 닿지 않음)이며, 이것을 고치지 않으면 5.2 의 스크롤 개선이 데스크톱에서 무의미해지므로 함께 수정했다 — `flexShrink: 0` 1개 속성.

### 5.6 회귀 가드 갱신

`tools/favorites_selftest.mjs` (`npm run fav:test`) 가 **기존 동작을 지키다가 실패**했다 — `items.slice(0,5)` 존재를 요구하는 검사가 있었고, 그 제한이 곧 BACKLOG-1 의 원인이었다.

판정 기준을 반전하고 신규 검사를 추가했다.

| 검사 | 변경 |
|---|---|
| 표시 개수 제한이 명시적(`slice` 존재) | → **잘라내지 않는다**(`slice` 부재) — 자르기 부활을 회귀로 차단 |
| — | 스크롤 컨테이너 존재(`maxHeight`+`overflowY:auto`) |
| — | 모바일 히트영역 44×44 |
| — | 패널 `flexShrink:0` (소멸 차단) |

결과: **18/18 통과**. `npm run nav:test` 36/36 통과(메뉴 구조 무변경 확인).
