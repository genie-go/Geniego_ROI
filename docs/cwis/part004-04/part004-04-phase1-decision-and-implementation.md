## 2. ★착수 직후 발견 — 즐겨찾기는 도달 불가능한 기능이었다

PHASE_3 구현을 위해 코드를 열자마자 **ST10의 완성률 50%가 과대평가**임이 드러났다.

```
toggleFav 호출부 전수 (초기 커밋 ~ 현재)
  Sidebar.jsx:391  QuickAccessPanel 의 ✕(해제) 버튼   ← 유일
```

즐겨찾기를 **추가하는 UI가 어디에도 없었다**. 연쇄 결과:

1. `favs`는 영원히 비어 있음 → 2. `hasFavs === false` → 3. QuickAccessPanel의 즐겨찾기 탭 자체가 렌더되지 않음 → 4. **해제 버튼조차 도달 불가**

`useFavorites` 훅도 `g_sidebar_favs` 저장 로직도 완벽하게 구현돼 있었지만 **한 번도 실행될 수 없었다**.

### 결정적 정황 — i18n은 이미 준비돼 있었다

```
sidebar.addFav  →  ko "즐겨찾기 추가" · en "Add to favorites" · ja "お気に入りに追加"  … 15개국 전부
```

번역이 15개국 전부에 존재하는데 그 키를 쓰는 버튼이 없었다. 설계 의도가 아니라 **미완성**이라는 강한 증거다. git 이력상 초기 커밋 이래 추가 버튼이 존재한 적이 없다.

### ST10 판정 정정

ST08은 `QuickAccessPanel`이라는 COMPONENT가 존재한다는 사실로 CAP-01(즐겨찾기 토글 UI)을 **present**로 판정했다. 그러나 증거축 검사는 *"컴포넌트가 존재하는가"*만 물었을 뿐 *"기능에 도달할 수 있는가"*는 묻지 않았다 — 전형적인 fake-looks-real이다.

**정정된 착수 시점 완성률: 50% → 33% (2/6)**

## 3. 구현 내역 (`frontend/src/layout/Sidebar.jsx`)

Golden Rule 준수 — 기존 `useFavorites` 훅·저장 키·`QuickAccessPanel`을 **교체하지 않고 확장**했다.

### 3-1. 즐겨찾기 추가 진입점 신설 (P0)

`FavStar` 컴포넌트를 신설해 **모든 메뉴 항목 옆에 별표 토글**을 배치했다(단일항목 섹션 + 다항목 섹션 양쪽).

- 미즐겨찾기 ☆(흐림) / 즐겨찾기 ★(노랑)
- `e.preventDefault()` + `e.stopPropagation()`으로 별표 클릭이 페이지 이동을 유발하지 않게 차단
- `<NavLink>` **내부가 아니라 형제**로 배치 — 앵커 안 버튼은 무효 HTML이며 클릭이 앵커에 먹힌다

### 3-2. 접근성 (CAP-04)

- 토글 버튼에 `aria-pressed={isFav}` — ST03 실측 `aria-pressed` 0건이었던 지점
- `aria-label`이 상태에 따라 "즐겨찾기 추가" ↔ "즐겨찾기 해제"로 전환
- 해제(✕) 버튼은 대상 이름을 포함: `"즐겨찾기 해제 — SMS 마케팅"`

### 3-3. 신규 항목 가시성 (CAP-05 부분)

`QuickAccessPanel`은 `[...favs]`(삽입 순)를 `slice(0, 5)`로 자른다. 즐겨찾기가 5개 있는 상태에서 6번째를 추가하면 **목록에 나타나지 않아 추가가 실패한 것처럼 보인다.** → `.reverse()`로 최신순 정렬해 방금 누른 항목이 항상 맨 위에 오도록 했다.

> **드래그 순서 변경은 구현하지 않았다.** 필요성 게이트 — 실사용 요구가 확인되지 않았고 ST10에서도 LOW·P3였다. 표시 5개 제한 하에서 사용자 정렬의 실익이 불분명하다. **CAP-05는 미완 Gap으로 남긴다.**

### 3-4. i18n

**추가 작업 없음.** `sidebar.addFav`·`sidebar.removeFav`가 15개국 전부에 이미 존재하며 정상 해석됨을 실측 확인했다.

두 사전이 다르다는 저장소 함정을 회피했다 — `NavSection`의 `t`는 `SIDEBAR_DICT`(평면 키)라 `sidebar.addFav` 같은 네임스페이스 키를 찾지 못한다. 라벨을 상위에서 `useT()`로 뽑아 `favLabels` prop으로 주입했다.

## 4. 브라우저 실측 검증 (데모 dev 서버, 오프라인)

`AuthContext.jsx:105`가 데모 모드에서 서버 401을 무시하므로 백엔드 없이 검증 가능했다.

| 검증 | 결과 |
|---|---|
| 별표 렌더 | **메뉴 64개 전부**에 버튼, `aria-pressed="false"`, 라벨 "즐겨찾기 추가" |
| 토글 동작 | 클릭 → `g_sidebar_favs = ["/auto-marketing"]` **최초 저장 성공** |
| 상태 전환 | `aria-pressed` → `true`, 글리프 → ★, 라벨 → "즐겨찾기 해제" |
| 패널 출현 | **"즐겨찾기 1" 탭이 처음으로 렌더됨** (이전엔 영원히 불가능) |
| 최신순 정렬 | 6개 추가 시 **마지막 추가분이 맨 위** — 수정 전이라면 안 보였을 항목 |
| 새로고침 영속 | 6건 복원, 별표 6개 `aria-pressed=true` |
| 해제 | ✕ 클릭 → 저장소 5건으로 감소 |
| **무후퇴** | 메뉴 링크 클릭 시 `/crm` **정상 이동** · 레이아웃·말줄임·행 높이 유지 |
| 콘솔 | 오류는 전부 가짜 토큰에 대한 API 401 — 본 변경과 무관 |

## 5. 회귀 테스트 신설 (CAP-06)

`tools/favorites_selftest.mjs` — `npm run fav:test`

14개 검사: **추가 진입점 존재(최우선 가드)** · FavStar 실배선 · props 전달 · aria-pressed · aria-label · 15개국 i18n · 최신순 정렬 · 저장 키 불변 · 테넌트 스코프 미사용 · NavLink 내 button 중첩 금지 · preventDefault.

### ★가드 실효성 실증

테스트가 실패할 수 있는지 확인했다 — `<FavStar>` 배선만 제거하자:

```
✗ FavStar 가 메뉴 렌더에 실제 배선됨 — 사용처 0곳
[favorites-selftest] 1건 실패
```

복원 후 전 항목 통과. **실패할 수 없는 테스트는 무가치하므로 이 확인이 핵심이다.**

### 작성 중 정정 1건

i18n 검사가 12개국을 누락으로 잡았다. 원인은 정규식이 `"addFav":`(따옴표)만 허용한 것 — `en.js`는 `addFav:`(따옴표 없음) 형식이었다. 파일마다 표기가 다르므로 양쪽을 모두 허용하고 `addFavorite`와 구분되도록 단어경계를 넣었다. **테스트 자신의 오탐이었다.**
