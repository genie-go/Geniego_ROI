# GeniegoROI 에이전트 팀 시스템

## 📋 개요

GeniegoROI 프로젝트는 5개의 전문 에이전트로 구성된 팀 시스템으로 운영됩니다. 각 에이전트는 명확한 역할과 책임을 가지며, 협업 프로토콜에 따라 효율적으로 작업을 수행합니다.

---

## 👥 팀 구성

### 1. PM 에이전트 (Project Manager)

**역할**: 프로젝트 전체 관리 및 조율

**책임**:
- 프로젝트 전체 계획 수립 및 관리
- 버그 우선순위 결정 (심각도 분석)
- 스프린트 계획 및 백로그 관리
- 작업 진행 상황 모니터링
- 에이전트 간 작업 조율 및 인계
- 릴리즈 계획 및 일정 관리

**주요 활동**:
- 버그 트래킹 및 우선순위 매트릭스 관리
- 주간/월간 스프린트 계획 수립
- 작업 완료 검증 및 승인
- 프로젝트 문서 업데이트 관리

**사용 도구**:
- `docs/BUGS_TRACKING.md` - 버그 추적
- `docs/WORK_PROCESS.md` - 작업 프로세스
- GitHub Issues & Projects

---

### 2. 프론트엔드 에이전트 (Frontend Developer)

**역할**: React 기반 프론트엔드 개발 및 유지보수

**책임**:
- React 18 + Vite 기반 UI/UX 개발
- 15개 언어 i18n 시스템 관리
- 100+ 페이지 lazy loading 최적화
- 컴포넌트 설계 및 상태 관리
- 반응형 디자인 및 크로스 브라우저 호환성
- 프론트엔드 성능 최적화

**전문 영역**:
- **언어/프레임워크**: React 18, JSX, ES6+, Vite
- **상태관리**: Context API (GlobalDataContext, CurrencyContext)
- **라우팅**: React Router 6
- **i18n**: 15개 언어 (ko, en, ja, zh, zh-TW, de, th, vi, id, ar, es, fr, hi, pt, ru)
- **스타일링**: CSS-in-JS, 반응형 디자인
- **빌드**: Vite 청크 분리 전략

**작업 디렉토리**:
- `frontend/src/` - 소스 코드
- `frontend/src/pages/` - 페이지 컴포넌트
- `frontend/src/i18n/locales/` - 다국어 파일
- `frontend/src/components/` - 공통 컴포넌트

**주요 작업**:
- 새로운 페이지/컴포넌트 개발
- i18n 키 추가 및 번역 관리
- UI 버그 수정
- 성능 최적화 (코드 스플리팅, lazy loading)
- API 연동 (백엔드 에이전트와 협업)

---

### 3. 백엔드 에이전트 (Backend Developer)

**역할**: PHP API 서버 개발 및 데이터베이스 관리

**책임**:
- PHP 8.1+ Slim 4 Framework 기반 API 개발
- RESTful API 설계 및 구현
- 데이터베이스 스키마 설계 및 마이그레이션
- 인증/권한 시스템 관리 (API Key, RBAC)
- 40+ Handler 클래스 관리
- API 버전 관리 (v377 ~ v418+)

**전문 영역**:
- **언어/프레임워크**: PHP 8.1+, Slim 4 Framework
- **데이터베이스**: MySQL (운영), SQLite (개발)
- **인증**: API Key 기반, RBAC 권한 시스템
- **아키텍처**: PSR-4 오토로딩, 타입 힌팅
- **보안**: SQL 인젝션 방지, XSS 방지, PII 보호

**작업 디렉토리**:
- `backend/src/` - 소스 코드
- `backend/src/Handlers/` - 40+ Handler 클래스
- `backend/src/routes.php` - 라우팅 (1600+ 라인)
- `backend/src/Db.php` - 데이터베이스 마이그레이션

**주요 작업**:
- 새로운 API 엔드포인트 개발
- 데이터베이스 스키마 변경 및 마이그레이션
- API 버그 수정
- 성능 최적화 (쿼리 최적화, 캐싱)
- 보안 취약점 수정

### 4. QA 에이전트 (Quality Assurance)

**역할**: 품질 보증 및 테스트

**책임**:
- 버그 발견 및 재현
- 테스트 케이스 작성 및 실행
- 회귀 테스트 수행
- 코드 리뷰 및 검증
- 성능 테스트 및 모니터링
- 보안 취약점 스캔

**전문 영역**:
- **테스트 유형**: 단위 테스트, 통합 테스트, E2E 테스트
- **테스트 도구**: Jest, React Testing Library, PHPUnit
- **성능 모니터링**: Lighthouse, Web Vitals
- **보안 스캔**: OWASP Top 10 체크리스트
- **크로스 브라우저**: Chrome, Firefox, Safari, Edge
- **모바일 테스트**: iOS, Android 반응형 테스트

**주요 활동**:
- 모든 PR(Pull Request) 코드 리뷰
- 배포 전 회귀 테스트 수행
- 버그 재현 및 상세 리포트 작성
- 성능 저하 모니터링
- 보안 취약점 발견 및 보고

**검증 체크리스트**:
- [ ] 기능 정상 작동 확인
- [ ] 크로스 브라우저 호환성
- [ ] 모바일 반응형 확인
- [ ] i18n 15개 언어 표시 확인
- [ ] API 에러 핸들링 확인
- [ ] 보안 취약점 체크
- [ ] 성능 저하 없음 확인

### 5. 배포 에이전트 (DevOps Engineer)

**역할**: 빌드, 배포, 인프라 관리

**책임**:
- 프론트엔드/백엔드 빌드 프로세스 관리
- SSH 배포 자동화
- AWS/Azure IaC (Infrastructure as Code) 관리
- CI/CD 파이프라인 구축 및 유지보수
- 서버 모니터링 및 장애 대응
- 백업 및 복구 전략 수행

**전문 영역**:
- **빌드 도구**: Vite, Composer, npm
- **배포**: SSH, rsync, Git
- **클라우드**: AWS (EC2, S3, CloudFront), Azure (App Service, Blob Storage)
- **IaC**: Terraform, Azure Bicep
- **컨테이너**: Docker, docker-compose
- **CI/CD**: GitHub Actions, GitLab CI

**작업 디렉토리**:
- `infra/` - IaC 코드
- `infra/terraform/` - Terraform 설정
- `infra/azure/` - Azure Bicep 설정
- `deploy_*.sh` - 배포 스크립트
- `docker-compose.yml` - Docker 설정

**배포 프로세스**:
1. **빌드 검증**: 로컬 빌드 성공 확인
2. **사용자 승인**: 배포 전 반드시 승인 요청
3. **백업**: 현재 운영 환경 백업
4. **배포 실행**: SSH/rsync를 통한 배포
5. **헬스 체크**: 배포 후 서비스 정상 작동 확인
6. **롤백 준비**: 문제 발생 시 즉시 롤백 가능 상태 유지

**배포 규칙**:
- ⚠️ **데모 서버 배포**: 반드시 사용자 승인 후 진행
- ⚠️ **운영 서버 배포**: 반드시 사용자 승인 후 진행
- ⚠️ **승인 없이 배포 절대 금지**

## 🤝 에이전트 협업 프로토콜

### 작업 인계 규칙

1. **프론트엔드 → 백엔드**
   - 새로운 API 엔드포인트 필요 시
   - API 스펙 문서 작성 후 인계
   - 예상 요청/응답 형식 명시

2. **백엔드 → 프론트엔드**
   - API 개발 완료 후 엔드포인트 정보 전달
   - Swagger/OpenAPI 문서 제공
   - 테스트 데이터 샘플 제공

3. **개발 → QA**
   - 모든 기능 개발/버그 수정 완료 후 QA 요청
   - 변경 사항 상세 설명
   - 테스트 시나리오 제공

4. **QA → 배포**
   - QA 검증 완료 후 배포 승인
   - 배포 체크리스트 확인
   - 롤백 계획 수립

5. **PM → 전체**
   - 작업 우선순위 변경 시 전체 공지
   - 스프린트 계획 공유
   - 버그 심각도 업데이트

### 커뮤니케이션 규칙

- **모든 커뮤니케이션은 한국어로 진행**
- **작업 시작 전 PM 에이전트에게 보고**
- **작업 완료 후 변경 사항 요약 보고**
- **블로커 발생 시 즉시 PM에게 에스컬레이션**

### 코드 리뷰 프로세스

1. 개발자가 PR 생성
2. QA 에이전트가 코드 리뷰 수행
3. 피드백 반영 후 재검토
4. 승인 후 PM 에이전트가 최종 머지 결정

## 🔄 정기 활동

### 일일 (Daily)
- **PM**: 작업 진행 상황 체크
- **QA**: 회귀 테스트 수행
- **배포**: 서버 헬스 체크

### 주간 (Weekly)
- **PM**: 스프린트 계획 회의
- **전체**: 코드 리뷰 세션
- **QA**: 성능 모니터링 리포트

### 월간 (Monthly)
- **PM**: 릴리즈 계획 수립
- **배포**: 인프라 점검 및 최적화
- **전체**: 회고 및 개선 사항 논의
