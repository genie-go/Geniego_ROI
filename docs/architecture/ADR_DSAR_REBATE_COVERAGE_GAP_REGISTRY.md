**측정 블록의 유일한 실패 방식은 좋은 숫자를 내는 것이다.**

## 결정

### D-1. 🔴 커버리지를 단일 %로 말하지 않는다

**Design / Implementation / Data / Verification 4축 분리. 종합 % 금지.**

**근거:** 06-A는 **설계는 두껍고(CANONICAL 90 · §53 101 · ADR 39) 구현은 0%**다.
합치는 순간 "설계 100%"가 "완료 100%"로 읽힌다 — **288차 가짜녹색의 문서판 재현**.

### D-2. 선행 문서를 인용하지 않고 재측정

1-1의 "Rebate 부재" 주장을 **믿지 않고 직접 grep**했다 →
**9/9 키워드(`rebate`·`scan-back`·`scanback`·`bill-back`·`billback`·`ship-and-debit`·`MDF`·`co-op`·`coop`)
`backend/src/` 히트 0** → **1-1 판정 유효 확인**.

**측정 블록이 측정 대상의 자기 보고를 인용하면 측정이 아니다.**

### D-3. 🔴 분모 부재 — 커버리지 계산 불가 (COV-GAP-01)

`grep "§53"` on 5-1 Canonical 2종 → **0**. **요구 목록이 저장소에 없다.**
스펙은 **채팅에만 존재**했고 **컨텍스트는 소멸**한다.

→ **`CoverageRequirementRegistry.source_persisted = false`인 요구는 분모에 넣을 수 없다.**
**세션 컨텍스트는 저장소가 아니다.**

> 이 규칙을 06-A에 소급하면 **현재 분모는 대부분 무효**다. 불편하지만 사실이다.

### D-4. 🔴 자기보고 불일치 — 52 vs 47 (COV-GAP-02)

`PM_CHANGE_HISTORY.md:294` "§53 52종" ↔ 실측 `ls DSAR_AUTHORIZATION_*.md` **47**
(`git ls-files` 동일 47 — **유실 아님**).

**대조군:** REBATE 측은 보고 49 ↔ 실측 49 **일치** → **집계는 가능했고 AUTHORIZATION만 어긋났다.**

**(a) 5건 누락 / (b) 과대 집계** — **D-3 때문에 판별 불가** → **`UNVERIFIED`**.
**누락으로 단정하지 않고, 괜찮다고도 하지 않는다.**

### D-5. 🔴 우선순위 = 운영영향 × 오신뢰 (Gap 크기 아님)

**ABSENT(Rebate 엔진 전체)가 최대 Gap이고 동시에 최하위 우선순위다.**

> **없는 기능은 아무도 해치지 않는다.** 고객이 못 쓸 뿐이다.
> **`Mapping.php:212`는 지금 운영에서 돌고, 1인이 두 번 누르면 정족수를 충족한다** —
> Maker-Checker가 **있다고 믿는 상태로 없다.**

**오신뢰가 결정적 인자다.** ABSENT는 아무도 믿지 않지만 **WIRING/DEFECT는 "있다"고 믿게 한다.**
**믿음이 있는 곳에서만 사고가 난다.**

### D-6. `UNVERIFIED`는 등재하되 등급 부여 금지

FP 레지스트리의 Gap 적용. **"모르면 P0"도 "모르면 무시"도 틀렸다** — **모른다는 것을 정확히 기록**한다.

### D-7. `GapAging` — 등재가 면죄부가 되지 않게

**Gap의 진짜 실패는 등재 실패가 아니라 방치다.** 임계 초과 → 5-7 Access Review 자동 상정.

### D-8. 측정 인프라 신설 금지 (Golden Rule)

`baseline.json`(sacred_sha — **기준선 동결 선례**) · **`render.mjs` 119 라우트 자동 도출**(281차 —
**분모 자동 산출 선례**)를 확장한다.

> **`render.mjs`가 신뢰받는 이유는 분모를 사람이 안 세고 코드에서 도출하기 때문이다.**
> **06-A는 정확히 그것을 안 했다** — 그래서 D-3/D-4가 발생했다.

| 대안 | 거부 사유 |
|---|---|
| 종합 커버리지 % 보고 | 설계 100% → 완료 100% 오독. **가짜녹색 문서판** |
