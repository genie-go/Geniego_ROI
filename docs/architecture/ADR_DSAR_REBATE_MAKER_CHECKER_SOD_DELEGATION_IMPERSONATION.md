## 결정 (핵심)

1. **★5-3 기술 정정 — 승인 2계통의 강도가 다르다**: `action_request` 는 `approvals_json` 을 갖지만 **정족수 판정에 쓰지 않는다** — **`$status = $decision === "approve" ? "approved" : "rejected";`**(Alerting.php:593) = **단일 승인 즉시 확정**. `approvals_json` 은 **이력 배열일 뿐**(:591). 즉 **정족수 REAL 은 `mapping_change_request` 하나뿐**이다(required_approvals DEFAULT 2). 5-3 이 두 계통을 동급 REAL 로 기술한 것을 정정한다.

2. **★5-3 §7 위임 2건 판정(자율·미확정·미수정)**:
   - **[판정 1] 중복 승인** — `$approvals[]` 에 dedup 없음(Mapping.php:212) → **동일인 2회 호출 시 required_approvals DEFAULT 2 를 혼자 충족 → approved**(:214). **자율 판정 = 결함 후보(의도로 보기 어려움)** — `DEFAULT 2` 의 설계 의도가 "**서로 다른 2명**"인 것은 명백하고, dedup 부재는 그 의도를 **무력화**한다(Quorum 을 둔 목적 상실).
   - **[판정 2] 자기 승인** — `requested_by`(:167)와 `$actor` 미비교 → **요청자 본인 승인 가능**. **자율 판정 = 결함 후보**(Maker-Checker 정의상 위반). 단 required_approvals=2 이므로 **판정 1 과 결합될 때만 단독 확정**(요청자 혼자 2회 승인 → approved = 가장 위험한 조합).
   - **★FP 규약상 PM 코드 재증명 전 P0 단정 금지**: 실 영향 판정 필요 = ①approve 엔드포인트 **호출 권한**(5-1 §PEP 100+ 분산) ②`mapping_change_request` **실 운영 사용 여부**(미사용이면 무증상) ③`self::actor()` 해석(동일인 식별 가능 여부). **본 세션 비파괴·미수정** → **MIGRATION_REQUIRED**.
   - **권장 해소**: **정족수 로직은 그대로 재사용**하고 **①동일 `user` 존재 시 거부**(`AUTHORIZATION_DUPLICATE_APPROVAL`) **②`exclude_requester=true`**(`AUTHORIZATION_SELF_APPROVAL_BLOCKED`)만 추가.
   - **★1인 테넌트 예외**: 레거시 단독회원(5-2 team_role fail-open 맥락)은 **승인자가 요청자뿐**일 수 있어 exclude_requester 가 **교착**을 유발할 수 있다 → **`allow_self_approve_when_sole_approver` 를 명시적 정책 + 감사 기록으로만 허용**(암묵 허용 금지).

3. **[관찰 3·신규] `action_request` 는 정족수가 없다** — 광고 예산/캠페인 액션을 **1명이 승인**. **의도 여부 불명**(policy_id 기반 자동화 액션의 경량 승인 설계일 수 있음) → **PM 재증명 대상** · 미수정. Rebate 금전 승인에 이 패턴을 **복제 금지**(SoD 문서 §10 Lint "정족수 없는 금전 승인").

4. **SoD Conflict Matrix(9쌍·자율 설계)**: **APPROVE_PAYOUT × EXECUTE_PAYOUT**(CRITICAL·5-1 §43 명시) · APPROVE_SETTLEMENT × EXECUTE_SETTLEMENT · **CREATE/UPDATE_VERSION × APPROVE/ACTIVATE**(1-4 §34·5-1 §34 "작성자≠활성화 담당자") · MANAGE_FUNDING × APPROVE · **ASSIGN_ROLE × APPROVE(자기 Role)**(5-2 SELF_ASSIGNMENT) · MANAGE_POLICY × VIEW/EXPORT_AUDIT · USE_PROVIDER_CREDENTIAL × ROTATE_CREDENTIAL · MANAGE_CLAIM × APPROVE_CLAIM · **IMPERSONATE × APPROVE/EXECUTE**(CRITICAL). **★현행 기반**: `TeamPermissions::ACTIONS` 에 **`approve` 와 `execute` 가 이미 별개**(:39) → **SoD 를 Action 수준에서 표현 가능**(Action 이 하나였다면 불가) · admin **master/sub 분리**(286차)도 SoD 인접.

5. **Toxic Combination**: 단일 Subject 의 위험 Role 조합(PAYOUT_APPROVER+PAYOUT_OPERATOR · ACCESS_ADMIN+FINANCE_REVIEWER · SECURITY_ADMIN+AUDITOR) — **5-1 §43 "동일 사용자에게 모든 고위험 권한 집중"=Critical Gap** 정합. **부여 시점(5-2)과 평가 시점(5-6) 양쪽 탐지**.

6. **SoD Enforcement Point 4단**: ①Assignment(5-2) ②Request ③**Decision(본 파트 핵심·현행 부재)** ④주기 점검(5-7). **★현행은 ③이 부재**(판정 1·2).

7. **SoD Exception**: **무기한 예외 금지**(valid_to 필수) · **상위 승인 + 보상 통제 필수** · 1인 테넌트 예외는 여기서 **명시 정책으로만**.

8. **★Impersonation 은 이 저장소에서 가장 잘 설계된 보안 기능 중 하나(정직)**: **관리자 가드**(requireAdmin→403·:474-475) · **관리자 대상 대행 차단**(대상 plan='admin' 이면 거부·:488 = **권한상승 차단**) · **토큰 분리**(`imp_` 접두·:493) · **2시간 만료**(:495) · **감사**(:499) · **사용자 배너 고지**(`_impersonated`·:525)가 **전부 REAL**. → **재사용·정본 승격**(중복 대행 경로 신설 금지).

9. **★Act-As 는 286차 사고의 산물이자 정본**: **"오직 admin 계정 + 오직 'platform_growth' 값만 허용(고객 테넌트 임의 임퍼소네이트 불가 = 블래스트 반경 최소) · 헤더 부재 = 정상 자기 테넌트(기본 OFF)"**(UserAuth.php:391-394). **★대행 기능의 최대 위험 = "자동 활성화 + 상태 고착"** — 286차: platform_growth **자동ON + localStorage 고착** → `authedTenant` 가 최고관리자 tenant 대신 platform_growth(데이터 0) 반환 → **전 메뉴 빈 화면**. **Blast Radius 6원칙**: ①대상 화이트리스트 ②**기본 OFF** ③**비영속** ④**요청 단위 해석**(세션 고착 금지) ⑤감사 ⑥고지.

10. **★[관찰·미확정] 대행 세션으로 승인·집행이 가능하다**: `imp_` 토큰이 `user_session` 에 **일반 세션과 동일하게 저장**(:496)되고 **대행 중 Action 제한이 없다** → 관리자가 **대행 세션으로 회원 대신 승인/집행 가능**(SoD Matrix 의 `IMPERSONATE × APPROVE/EXECUTE` = **CRITICAL**). **단 대상이 admin 이면 대행 자체가 차단**(:488)이라 **블래스트 반경은 일반 회원 범위**. **FP 규약상 PM 재증명 전 P0 단정 금지** · **비파괴 미수정** → **§4c Action Restriction 신설로 해소 권장**.

11. **Impersonation Action Restriction(신설·관찰 반영)**: 대행 세션 **기본 = 읽기 전용** — 현행 엔드포인트명이 **"회원세션(관리자 **대행 열람**)"**(:466)이므로 **설계 의도와 정합**. 금지: **APPROVE/EXECUTE 계열 전면**(승인·정산·지급·집행) · ASSIGN_ROLE · MANAGE_POLICY · ROTATE_CREDENTIAL · USE_PROVIDER_CREDENTIAL · 결제/구독 변경 · 데이터 삭제 · EXPORT(또는 REQUIRE_APPROVAL).
