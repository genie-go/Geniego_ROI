## 결정 (핵심)

1. **★tamper-evident 감사가 이미 REAL(정직·최대 발견)**: **`menu_audit_log`(N-152-A)** 가 **`hash_chain CHAR(64)` + old/new value JSON + changed_by + changed_by_role + reason + ip_address + user_agent + request_id** 를 갖춘 **감사 정본**이며, **직전 해시를 조회해 실제 체인을 계산**한다(`SELECT hash_chain ... ORDER BY id DESC LIMIT 1`·AdminMenu.php:216). **SIEM Export 도 REAL**(LEEF 2.0 · RFC 5424 Syslog·282차).

2. **★[선행설계 R3 기술 정밀화] "hash-chain 부재"는 부정확했다**: 선행설계 **R3(Accrual/Ledger) §49** 가 **"hash-chain Ledger 부재(4-5-1-1 확정) → 신설"** 이라 기술했으나, **hash-chain 패턴은 REAL(감사 로그·menu_audit_log·N-152-A)** 이고 **금전 원장에만 부재**다. → **R3 의 hash-chain 은 "신설"이 아니라 "menu_audit_log 패턴을 금전 원장으로 확장"** 이 정합(Golden Rule=Extend · **신규 hash-chain 구현 금지**). **본 블록에서 R3 기술을 정밀화**.

3. **★[설계 순환 참조 — 자기 지적] 전 블록이 존재하지 않는 기능에 의존했다**: 5-1 §43 이 Critical Gap 18종의 대응을 **"Access Review 차단"** 으로 정의했고 5-2·5-3·5-4·R1~R5 가 이를 인용했다. **그러나 Access Review 는 부재(grep 0)** — **존재하지 않는 기능에 의존한 설계**였다.
   - **★해소(자율 판단)**: "Access Review 차단"의 실질은 **①즉시 차단(Runtime Guard·5-6 PDP)** = 지금 이 요청을 막는 것 · **②사후 검토(Access Review)** = 왜 그런 권한이 있었는지 주기 점검 — **두 가지로 분해**된다. **Critical Gap 의 1차 대응은 ①(Runtime Guard)이어야 하고 Access Review 는 2차(재발 방지)**. **앞선 블록의 "Access Review 차단" 표기는 "Runtime Guard 차단 + Access Review 등재"로 읽어야 한다** — 본 ADR 에서 정정.

4. **감사 스키마 편차 — KEEP_SEPARATE 유지 + 표준 승격(자율 판단)**: `audit_log`(**5필드**: id·actor·action·details_json·created_at) vs `menu_audit_log`(**12필드+hash_chain**). **5-1 이 "audit_log 12파일 = 도메인별 KEEP_SEPARATE(스키마 상이)" 로 판정**했으나 **본 블록 재검토**: **스키마 차이의 원인이 "도메인 요구"가 아니라 "구현 시점·투자 차이"로 보인다**(menu_audit_log=N-152-A 전용 설계 · audit_log=범용 간이). → **★KEEP_SEPARATE 는 유지하되(테이블 통합은 회귀 위험) `menu_audit_log` 를 감사 표준 스키마로 승격하고 신규 감사는 그 수준을 따른다**(**하향 평준화 금지** · 점진 상향은 멱등 ALTER 패턴). **PM 재증명 전 확정 금지**.

5. **★[관찰·미확정] `audit_log` 에 tenant_id 가 없다**(AdminGrowth.php:158) — 멀티테넌트 감사 조회 시 **테넌트 격리 불가**. AdminGrowth 가 **플랫폼 관리자 도메인이라 의도일 수 있으나** 확인 필요(실 확인=이 테이블이 테넌트별 조회에 쓰이는지·플랫폼 전용인지). **FP 규약상 PM 재증명 전 P0 단정 금지** · **비파괴 미수정**.

6. **표준 스키마(§1)**: id · **tenant_id**(★관찰 5 반영) · resource_type · resource_id · action · **old_value/new_value**(민감 원문 금지→Reference) · **actor · actor_role** · **reason** · ip_address · user_agent · **request_id**(5-1 Context 상관관계) · **hash_chain** · created_at. **신규 감사는 표준 이상** · **기존 테이블 통합 금지**(KEEP_SEPARATE).

7. **Hash Chain(§2)**: **현행 정본 재사용**(menu_audit_log) · **검증 Job 은 부재 → 신설**(체인 단절 = `AUTHORIZATION_AUDIT_CHAIN_BROKEN` Critical) · **적용 확대**(Rebate 금전 원장 · Authorization Decision · Role Assignment · 승인 · Break Glass). **★한계 정직**: hash chain 은 **append-only 무결성**만 주고 **전량 삭제·테이블 교체는 못 막는다** → **SIEM 외부 반출과 병행해야 실효**(§5).

8. **Authorization Decision 감사(§3) — 볼륨 정책(자율)**: **1,448 라우트 × 전 요청** 감사는 폭증 → **①ALLOW 는 샘플링/집계 ②DENY 는 전량 ③고위험(금전·Credential·PII·Production)은 전량**(5-1 §30 `LOG_SENSITIVE_ACCESS`) **④CONDITIONAL/STEP_UP/MASKING 전량**. **★DENY 를 샘플링하면 안 된다**(공격 탐지 불가). **"감사 실패 시 Action 차단"은 고위험에 한해**(전 요청 적용 시 가용성 위험).

9. **Retention·SIEM(§4·§5)**: 보존기간 정책 **부재 → 신설** · **금전/세무 감사는 법정 보존**(R5 "Erasure=익명화+보존" 정합) · **DSAR 삭제 요청에도 감사 원장 물리 삭제 금지**(PII 만 익명화) · **아카이브 경계에서 hash chain 연속성 유지**. **★SIEM 은 282차 정본 재사용**(LEEF 2.0 · RFC 5424) — **중복 Exporter 신설 금지** · **Credential/Token/PII 원문 반출 금지**(5-1 §48).

10. **Access Review(§2·짝 문서)**: Campaign(trigger: PERIODIC / **CRITICAL_GAP_DETECTED** / INCIDENT / ROLE_CHANGE / SOD_VIOLATION / AUDIT_FINDING) · Item(부여 근거 — **★5-2 에서 assigned_by/reason/approval 이 부재 확정이므로 "근거 없음"이 다수일 것**) · Decision(CERTIFY/REVOKE/MODIFY_SCOPE/REQUIRE_JUSTIFICATION/ESCALATE) · **★REVOKE 는 실제 회수까지 확인**(5-5 "권한 회수 = Grant REVOKED + 세션 종료 + 캐시 무효화" — **Review 가 REVOKE 했는데 권한이 남으면 Review 는 무의미**) · **Reviewer ≠ subject**(5-4 Maker-Checker 계승) · **Certification 은 무기한 유효 아님**(next_review_at 필수).

11. **Orphan·Dormant(§3·§4) — 원재료는 이미 있다**: **api_key `last_used_at`·`use_count`**(Db.php:950-952)로 **Dormant 키 탐지 가능**(5-1 §37 "미사용 Account 비활성화" 를 **새 필드 없이** 구현). **★단 인간 Subject(app_user)에는 `last_used_at` 부재** → **세션 이력 기반 추정** 필요. **Orphan 은 SCIM `active===0` → 세션 즉시 삭제**(EnterpriseAuth.php:400)가 **IdP 연동 시 자동 해소** · **비 SCIM 경로는 Review 대상**.

12. **★[5-5 위임 판정·자율] 만료 세션 정리 Job 부재 = 위생 이슈(기능 결함 아님)**: `DELETE FROM user_session` 는 **deprovision·강제종료 경로만**이고 주기 정리가 없어 **만료 행 누적**. **판정**: 만료는 **조회 시점 검증**으로 처리되므로 **보안 영향 없음**. 부작용 = 테이블 비대 · 만료 토큰 DB 잔존(단 검증으로 무효) · Review 분석 노이즈. **권장**: **주기 cron 정리**(현행 cron 인프라 재사용 — SMS/Kakao 예약 드레인 워커·shelf_rank_cron·media_gc_cron 선례) · **삭제 전 감사 집계**.

13. **Compliance Mapping(§7)**: framework(내부 헌법 / GDPR / 개인정보보호법) ↔ Control ↔ Evidence 위치 ↔ 검증 방법. **현행 재사용**: Compliance.php(SIEM) · GdprConsent · Dsar(5경로) · 헌법 Vol1~5(Trust READY·Explainable·No-PII·테넌트 격리) · baseline.json. **★법률 자동 확정 금지** → `LEGAL_REVIEW_REQUIRED`(선행 Privacy 파트 계승) · 본 문서는 **프레임워크만 제공**.
