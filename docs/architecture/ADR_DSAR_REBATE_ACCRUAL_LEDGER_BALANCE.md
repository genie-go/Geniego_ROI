## 결정 (핵심)

1. **★실측 정직·Rebate Accrual 엔진 부재·인접 재사용**: **Rebate Accrual/Ledger/Balance 엔진은 부재(NOT_APPLICABLE·grep 0)**. 그러나 **완성된 금전 원장 거버넌스 선례가 `ad_spend_ledger`에 존재**(278차 이중청구 3중 방어): 상태기계(pending→charging→charged/reconciling)·UNIQUE 선점·결정적 Idempotency Key·고아 선점 회수·외부 원장이 진실·정수 금액(BIGINT)·정직 pending 기록. 역분개 멱등=ChannelSync(`CHS-`/`CHR-`). **기존 패턴 재사용(중복 원장/집계 엔진 신설 금지·§40)**·hash-chain 부재(신설). 지어내기·NO_DATA/오탐 금지·전방호환 계약.

2. **Accrual≠Availability≠Claim≠Settlement≠Payout(§38·§4.1)**: 발생·가용·청구·정산·지급은 각각 별개 단계(4-5-3-1-6 §4.2 "Qualification≠Accrual≠Availability" 위임 수령). Accrual은 **pinned program/rule/tier version + qualification evaluation reference + funding allocation/commitment reference**를 필수 보유. **Source 없는 Accrual 생성 금지(근거 없는 금전 발생 금지)·동일 Source 다중 Accrual 귀속 금지**(4-5-3-1-3 §4.10 Double Funding 대칭).

3. **★Ledger=Append-only·Immutable(§40·§41·§4.2)**: 기존 Entry **UPDATE/DELETE 금지** — 상태 전이만 허용하고 전이도 **원자적 조건부 UPDATE+rowCount·단일 프로세스**("회수 자체도 원자적 상태전이로 단일 프로세스만 수행"·BillingMethod.php:549). **true-up/정정/역분개=신규 Entry**(4-5-3-1-5 §22b·4-5-3-1-4 §26b·§32 계승).
