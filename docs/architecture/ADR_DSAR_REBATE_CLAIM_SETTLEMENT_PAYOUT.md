## 결정 (핵심)

1. **★실측 정직·Rebate Claim/Settlement/Payout 엔진 부재**: **전부 부재(NOT_APPLICABLE·grep 0)** — 특히 **outbound 지급 경로 자체가 이 저장소에 없다**(`payout` 히트는 전부 inbound `net_payout`·Toss/Paddle은 inbound 수납). 실 인접=Claim 멱등 적재(recordClaim `CLM-`·**도메인 상이·패턴만**)·**Estimated≠Confirmed(OrderHub)**·**가짜집행 근절 3-state 정직(Alerting 287차)**·**자격증명 게이트(AdAdapters `no_credentials`)**·**실패 시 상태 미변경(AutoCampaign 233차)**·결정적 Key/UNIQUE/고아 회수/외부 원장 진실(BillingMethod 278차). 지어내기·NO_DATA/오탐 금지·전방호환 계약.

2. **Claim≠Accrual·Approval≠Settlement≠Payout(§63·§67·§4.1·§4.2)**: Accrual(우리 산출)≠Claim(참여자 주장) — **Accrual 없는 Claim=검증 대상(자동 거절 아님)·Claim 없는 자동 지급 여부는 Program 정책(묵시적 가정 금지)**. **승인은 지급이 아니다**(4-5-3-1-4 §4.5·선행설계 R2 §4.2 계승)·승인 시 Ledger CLAIM_LINK Entry·**Availability 전환은 Holdback/Maturity 통과 후**(선행설계 R2 §20d/§21). Claim Model/Settlement Method는 **4-5-3-1-2 분류축 연결(재정의 금지)**.

3. **Validation fail-closed(§66)**: Rule(12·DUPLICATE·ELIGIBILITY·QUALIFICATION·**ACCRUAL_MATCH**·SCOPE·DEADLINE·EVIDENCE·CALCULATION_RECHECK(as-of Version)·FUNDING/COMMITMENT·FRAUD·CONTRACT·FX). **★UNKNOWN=자동 승인 금지·검증 근거 없는 승인 금지·Provider 리포트 무검증 신뢰 금지**(헌법 Vol3 Cross Validation). 부분 승인 지원(approved_amount≠claimed_amount)·**거절 근거 통지 필수**·**분쟁 중 지급 보류·분쟁 해결도 Append-only(조용한 금액 수정 금지)**.

4. **★Estimated≠Confirmed(§68b·§4.3 — OrderHub 정본 승격)**: **"실 정산이 이미 있으면 추정 스킵(정합 보존)"**(`$exStatus !== 'estimated' → continue`·OrderHub.php:1268-1271)·**zero-out도 estimated 행만**(:1297/1305)·프로그래매틱 적재 기본 `'confirmed'`(:1121). **★Rebate: 추정 정산은 실 정산 도착 시 보존/스킵·추정을 확정으로 표기 금지·추정 단계 비용/수수료 날조 금지**("실제 취소수수료는 실 정산 ingest가 반영"·:1233 정신 계승·**임의 숫자 금지 원칙 정합**). Settlement Line 차이는 **조용히 맞추기 금지(Discrepancy·선행설계 R3 §60)**.

5. **★★가짜집행 금지 — 본 파트 최우선 계약(§72·§4.4)**: **현행 정본이 사고를 명시** — "종전엔 `status='executed'` 로만 바꾸고 **어떤 외부 API 도 호출하지 않아** … 실제 광고 캠페인은 그대로 소진되던 **fake-looks-real**"(Alerting.php:608-609·287차) → **"집행 결과에 따라 상태를 정직하게 기록(`executed`=실집행성공 / `failed`=실패 / `approved_manual`=자동집행 불가)"**(:610-611). **288차 ChannelSync 하드실패 `ok=>true` 위장(14채널 18개소)도 동일 클래스** — **이 저장소가 두 번 사고 낸 재발 클래스**이고 **Payout은 돈이 실제 나가는 경로**라 피해 최대. **★EXECUTED 전이는 ①외부 호출 ②응답 수신 ③성공 판정 ④Provider Response 근거 저장을 모두 충족해야만 허용**·**CI 가드(§79) 대상 지정**.

6. **정직 State 8종·미집행≠실패(§72c·§4.5)**: PENDING·**NOT_EXECUTED_NO_CREDENTIALS**·**NOT_EXECUTED_NOT_VERIFIED**·**NOT_EXECUTED_NO_FUNDING**·**APPROVED_MANUAL**·**EXECUTING**(in-flight·★선행설계 R3 §53 잔액 포함 필수)·**EXECUTED**(실 집행 성공만)·**FAILED**(호출 후 거절). **★현행 정본**: AdAdapters **"자격증명 없으면 `no_credentials` 반환(실 API 호출 안 함)"**(:19)=**미집행의 정직한 표기(실패 아님)**·**"★미검증 고지: 라이브 검증 후 운영 권장"**(:25)=**문서 스펙 기준 구현을 검증 완료로 표기 금지**(288차 가짜녹색 정합).

7. **실패/불명 시 발산 금지(§72b)**: **현행 정본** AutoCampaign kill-switch가 **push 실패 시 DB 상태 미변경·502**(:602-609). **★타임아웃/네트워크 오류 등 결과 불명 시 EXECUTED/FAILED 임의 단정 금지 → `EXECUTING` 유지 후 외부 조회 정합**(고아 회수·선행설계 R3 §46·"방치된 선점 회수를 '잔액 계산 전에' 수행" 순서 계승).
