## 결정 (핵심)

1. **★실측 정직·Rebate Recovery 엔진 부재·인접 재사용**: **Rebate Recovery/Clawback/Overpayment/Write-off 엔진은 부재(NOT_APPLICABLE·grep 0)**. 그러나 **사후 회수의 실 정본이 존재**: Type 분류(Paddle `full|partial|chargeback`)·**partial 과다역분개 방지**·**Entitlement 회수(setUserPlan 'demo')**·**원 거래 기간 귀속(원주문 월 재롤업)**·원 거래 금액 기준·멱등 1회·**외부 원장 확인 후 정정**·**"매입사 원장이 진실"**. 지어내기·NO_DATA/오탐 금지·전방호환 계약.

2. **Recovery≠Reversal≠Disqualification(§80·§4.1·§4.2)**: **Reversal**(4-5-3-1-7 §47)=미확정/오기록 원장 정정(부호 반대 Entry)·**Recovery**=확정·지급된 금전을 되받는 실행(외부 상대방 관여·거절 가능·시효 존재)·**Disqualification**(4-5-3-1-6 §23 위임 수령)=**미확정 단계 Availability 차단(음수 지급/회수 아님·Referral 영구 잠금 정본)**. **확정 지급 후에만 Recovery**. Trigger(12·**Trigger/Cause 없는 회수 금지**)·Type(6).

3. **★Partial≠Full — 과다역분개 금지(§83·§4.3·Paddle 272차 정본 승격)**: **"full/chargeback 환불 → MRR 역분개(partial 은 서비스 유지라 제외)"**(Paddle.php:738) — **부분 회수에 전체 효과를 적용하면 과다역분개**. **★Rebate: 부분 회수 시 ①금액만 부분 차감 ②Entitlement/Tier/파생 지표는 재평가 후 적용(자동 전체 취소 금지) ③Tier 하향은 재계산 결과로만 판정**(4-5-3-1-5 §22a as-of). 4-5-3-1-3 "부분클레임 과다역분개" 정합.

4. **★회수 금액·기간 기준(§83b·§84·§4.4·ChannelSync 정본 승격)**: **금액=원 거래/원 지급 기준**(`$claimTotal = $existing['total_price']` 원주문 우선·:4403-4404)·**★현재 Rule/Version으로 회수액 재계산 금지(as-of 원 지급 시점 Version)·회수 시점 환율 재계산 금지(원 지급 rate version·차액은 FX_ADJUSTMENT Entry)**(4-5-3-1-4 §32·4-5-3-1-5 §32 계승). **★기간=원 거래 기간에 귀속** — **현행 정본: 원주문 월을 재롤업 대상으로 표시**(`$am = substr($existing['ordered_at'],0,7)`·`$affectedMonths[$am] = true`·:4401-4402). **회수를 당월 비용으로만 처리하면 원 기간 리베이트가 과대 잔존**(268차·263차 역분개 정신 계승)·회계 마감 후 회수는 Prior Period Adjustment 여부 명시.

5. **Entitlement 회수(§85·§4.6)**: **현행 정본** full/chargeback 시 **`setUserPlan($db, $email, 'demo', null)`**(지급된 플랜 권한 즉시 회수·Paddle.php:737). **★금전 회수와 혜택 회수는 별개 축**(금액만 회수하고 Tier 혜택 유지=손실·역도 성립)·**단 §4.3에 따라 부분 회수 시 자동 전체 혜택 회수 금지**.

6. **★Chargeback≠Dispute(§91·§4.5·Paddle 정본)**: **`$isFull = in_array($action, ['full','chargeback'], true)`**(Paddle.php:724)가 **chargeback을 full 환불과 동일하게 즉시 반영**(분쟁 대기 아님). **★Rebate: chargeback은 외부(카드사/은행)가 이미 강제 집행한 기정사실 → 원장 즉시 반영(Append-only)·"분쟁 중"으로 보류 금지·이의 제기는 별도 Dispute 병행**. 반면 **진행중 쟁송(Dispute)은 §4.11에 따라 해당 건 지급·회수 보류**(4-5-3-1-6 §19·4-5-3-1-8 §67c 계승).

7. **★예방>회수(§90b·§4.12)**: **과지급은 사후 회수가 아니라 지급 전 3층 방어로 막는 것이 정본**(4-5-3-1-7 §44/§53 위임 수령) — ①결정적 Payout Key(**microtime 금지**·"Toss 가 최후의 방어선") ②UNIQUE(settlement point) 선점 ③**in-flight 포함 잔액 집계**(**"빠지면 … 같은 delta 를 다시 청구 = 이중청구의 원인이었다"**·BillingMethod.php:445-447). **★회수 로직 신설이 예방 부재를 대체하지 못한다**. Cause(10)·**Cause 미규명 회수 금지(재발)·Cause별 재발 방지 조치 기록**.

8. **★외부 원장 확인 후 정정(§92·§4.11·BillingMethod 정본)**: **추정으로 회수 금지** — **현행 정본**: 고아 선점 회수 시 **Toss 주문조회로 "실제로 돈이 나갔는지" 매입사 원장에서 확인 후 정정**(:476/536)·**"우리 DB 가 아니라 매입사 원장이 진실이다. 회수 자체도 원자적 상태전이로 단일 프로세스만 수행"**(:549). **★Rebate: 과지급 의심 시 외부 지급 원장에서 실 이체 확인 후에만 Recovery 개시 — 미이체 건을 회수하면 이중 손실**.

9. **Method·Netting·시효(§86·§86b·§87)**: **★outbound라 PG refund API를 쓸 수 없다 → Netting(차기 지급 상계)·Debit Memo가 현실적 주 수단**(§0 관찰·**Paddle refund 경로 복제 금지**). Netting=**상계도 Ledger Entry·양방 동시 반영(4-5-3-1-8 §70)·한도 필수(지급액 음수 금지)·무통지 상계 금지**. **시효(계약/법정)·한도·소액 면제·시효 경과 시 WRITE_OFF 전환·★시효 경과 후 회수 시도 금지·시효 정책 미지정 시 회수 개시 금지(fail-closed)**.

10. **★가짜 회수 금지·Write-off≠회수 성공(§88·§96·§4.9·§4.10)**: **4-5-3-1-8 §72 정직 State 계승** — **외부 회수 미실행인데 RECOVERED 표기 금지**(fake-looks-real·287/288차 재발 클래스). State(8: PENDING·NOT_EXECUTED_NO_METHOD·NETTING_SCHEDULED·EXECUTING(**in-flight 잔액 포함**)·**RECOVERED(실 회수 확인만)**·PARTIALLY_RECOVERED·FAILED·**WRITTEN_OFF**). **★Write-off=손실 확정·회수 성공 아님 → RECOVERED 표기 금지·상위 승인+사유+한도(남용 방지)**. 회수도 **Append-only Entry(CLAWBACK_REFERENCE·원 Payout Entry 삭제/수정 금지)**·**멱등**(ChannelSync 전이 1회·`CLM-`/`CHR-` 계승·중복 회수 금지)·**무통지 회수 금지·분쟁 제기 경로 안내**.

11. **Dispute·Evidence·Resolution·Closure(§93·§94·§97)**: Dispute State(8)·Claim Basis(9·**UNFAIR_RECOVERY** 포함)·**Evidence Package=계약 Reference(원문 복제 금지)+version snapshot+evaluation trace(입력/임계/연산자/매치 경로)+ledger entries+hash-chain 검증**. **★이것이 앞선 8개 파트가 Evidence/Trace/Version pin/hash-chain을 강제한 이유 — 분쟁 시점에 "왜 이 금액인지" 재현 가능해야 한다**(헌법 Vol4 Explainable).
