## 결정 (핵심)

1. **★통합이지 신설이 아니다(§51)**: **VALIDATED_LEGACY 6종을 Canonical Foundation 으로 승격** — ①`TeamPermissions::ACTIONS`(8) → Canonical Action 기반 ②`TeamPermissions::DATA_SCOPES`(9) → Canonical Scope 기반 ③`acl_permission`(메뉴×8동작) → Permission Registry 기반 ④api_key RBAC(roleRank 4+scopes_json) → API_CLIENT Subject/Role 정본 ⑤`PlanPolicy`(RANK+기능키→최소플랜) → Policy Registry 기반 ⑥Tenant Isolation(auth_tenant 주입·authedTenant 64·RLS·IDOR 차단) → 재사용. **★Rebate 전용 IAM 신설 금지 · 이미 3계통인 Role 을 4번째로 늘리지 말 것**(헌법 Golden Rule=Extend · `feedback_no_duplicate_features`).

2. **★현행 부재 확정(신설 대상)**: **중앙 PDP · Authorization Decision 기록 · Policy Version/Immutable Hash · Obligation · Policy Conflict 해소 · Reconciliation · ABAC Context(device/network/risk/session_age/amount) · Field Access Profile · Break Glass · Access Review · Production/Sandbox 권한 분리**. **있다고 가정한 배선 금지**(287차 죽은 스켈레톤 교훈) → NOT_APPLICABLE 표기 + Reference 필드만.

3. **Authentication≠Authorization(§4.7)**: 현행 인증(UserAuth 세션·MFA·EnterpriseAuth SAML/OIDC/SCIM·OAuth)은 **KEEP_SEPARATE**. **로그인 성공/유효 Access Token 만으로 Rebate Resource 접근 허용 금지**.

4. **Subject 통합(§7·§8)**: Subject Type(21). **현행 4계통을 Canonical Subject 로 통합** — app_user(team_role) · **api_key=API_CLIENT**(key_hash SHA-256·scopes_json·expires_at) · partner_session · agency `agt_`(서버바인딩 tenant·위조불가). Binding(17 축)에 Tenant/Workspace/Team/Legal Entity/**Environment**/**Provider Account**/Contract/**Rebate Program** 연결. **Service Account·Machine Identity 는 인간과 별도 Subject Type · Human Login·Impersonation 금지**(§37).

5. **Resource·Action 등록(§9~§11)**: **Rebate Resource Type 40**(Program Foundation 7 · Funding·Financial 8 · Operational 12 · Infrastructure 9) 등록. Action 은 **현행 8동작을 기반으로 확장** — **`approve`·`execute` 가 이미 분리**되어 있어 §43 "Financial Approval 과 Execution Permission 결합=Critical" 방지의 실 근거가 된다. Resource Attribute 는 **Decision 시점 Snapshot**(현재 값으로 과거 Decision 근거 덮어쓰기 금지·4-5-3-1-4 §38 정합).

6. **Permission·Granularity·Field Access(§12~§14)**: Permission=Resource Type×Action×Scope Template×Condition Template×**Field Access Profile**×Effect. **`acl_permission` 매트릭스에 Rebate Resource×Action 등록**(중복 Permission Store 금지) · scopes_json 은 API_CLIENT Bundle 로 병행(KEEP_SEPARATE). Granularity 20 중 **Tenant/Row/메뉴 Operation 은 REAL**, **Instance/Program/Legal Entity/Environment/Provider Account/Financial Threshold/Field 는 신설**. **Field Access Profile(대상 22·Level 8)** — **현행 Masking 이 3+ 곳에 산재 → CONSOLIDATION_REQUIRED**(중복 Masking Engine 금지) · **Sensitive Field 기본 MASKED/TOKENIZED_REFERENCE**.

7. **Role≠Scope(§4.9·§15~§18)**: **Role 이름으로 권한 하드코딩 금지 — Role-Permission Binding 사용**(§44 Lint). `REBATE_PROGRAM_MANAGER` 보유해도 **허용 Scope 내에서만**. Role 3계통(team_role·api_key role·admin master/sub)은 **CONSOLIDATION_REQUIRED**이되 **실효 동작 보존(Legacy Equivalence=5-8)**. Subject-Role Assignment 에 Tenant/Workspace/Legal Entity/Environment/Program Scope/Validity/assigned_by/Evidence 기록.

8. **RBAC·ABAC·PBAC 조합(§5)**: 경쟁 모델 금지 · 단일 Canonical Contract. **현행 매핑**: RBAC=REAL(3계통) · ABAC=부분(tenant/plan REAL · legal_entity/country/device/risk 부재) · PBAC=부분(PlanPolicy 만).

9. **Policy Version·Historical(§22)**: **Approved Policy Version 은 Immutable Hash + Effective Period 보존** · **현재 Policy 로 과거 Authorization Decision 근거 덮어쓰기 금지**(4-5-3-1-4 §4.3/§38 계승). **현행 부재→신설**.

10. **Deny-by-default·Explicit Deny 우선(§4.1·§4.2·§32·§33)**: 명시적 Allow 없으면 거부 · Allow/Deny 충돌 시 **Explicit Deny 우선**(Break Glass/Emergency 는 5-5 전용 정책으로만 예외). 평가 순서 18(**Tenant/Environment Isolation 을 Role 평가보다 먼저**). Conflict Strategy 7(EXPLICIT_DENY_WINS·RESTRICTIVE_EFFECT_WINS 등) · **Production 미해결 Critical Conflict = 접근 차단**. **Condition 의 `null_behavior`/`stale_behavior`/`failure_effect` 필수 = 기본 DENY** · **Operator 화이트리스트 강제·임의 eval 금지**(현행 정본: RuleEngine 미등록 op **422 거부**·RuleEngine.php:120-121).

11. **Decision 추론 가능(§4.10·§29)**: Decision 에 subject·resource·action·roles·permissions·**policies·policy_versions**·matched/unmatched conditions·explicit_deny·scope·field access·auth·risk·**final_effect·denial_reason·obligations** 기록(헌법 Vol4 Explainable). **현행 부재** — 판정 근거를 남기는 구조가 없다(신설). Obligation 17(MASK_FIELDS·LIMIT_ROWS·LIMIT_AMOUNT·REQUIRE_APPROVAL·REQUIRE_MFA·DISABLE_CACHE 등).

12. **UI·API 일관성(§4.8) — ★현행 실 위험**: **`PlanPolicy` ↔ 프론트 `planMenuPolicy.js` 수동 동기화**("변경 시 양쪽 동시 갱신"·PlanPolicy.php:14)이며 **286차에 rank 맵 붕괴 사고**(starter=growth=pro=1 → requirePro 가 사실상 'starter+'·UserAuth.php:330)로 **드리프트가 실현된 이력**이 있다. → **UI_API_MISMATCH Reconciliation 대상** · 실 통합은 **5-6**. **UI 메뉴 숨김만으로 보호 금지 — API/Service/Database 까지 동일 Decision Contract**.

13. **Provider Account·Credential 분리(§39)**: Metadata 조회 / 사용 / Rotation / Disable 을 **서로 다른 Permission**. **Credential Secret 원문 조회 권한 생성 금지 · Tokenized Runtime Reference 로만 사용**. **현행 재사용**: channel_credential **AES-256-GCM**(267차) · ChannelCreds 마스킹 · **`no_credentials` 게이트**(실 API 호출 안 함·AdAdapters.php:19).

15. **★인접 관찰 3건(코드변경 0·근거 기록만)**: ①**인가 판정 100+ 지점 분산·중앙 PDP 부재**(authedTenant 64 · requirePro/requirePlan 56/호출부 351 · requireMasterAdmin2 5 · 미들웨어 1) — §4.8 구조적 위험 · **275차 헤더리스 getJson 401 회귀 2차 재발**이 실현 사례 → **5-6 통합**. ②**PlanPolicy↔UI 수동 동기화**(286차 사고) → Reconciliation. ③**`team_role` fail-open**(미설정=owner·AdminMenu.php:52-54) — **§4.1 Deny-by-default 와 상충**하나 레거시 단독회원 호환을 위한 **의도적 설계로 보임** · **FP 레지스트리 규약상 PM 코드 재증명 전 P0 단정 금지** · 비파괴 미수정 → **5-2 에서 정책 판정**(`MIGRATION_REQUIRED` 후보).
