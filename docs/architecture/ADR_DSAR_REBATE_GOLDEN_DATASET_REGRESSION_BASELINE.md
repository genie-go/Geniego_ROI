## 결정

### D-1. 🔴 분리 판정 — 1-7처럼 한 마디로 덮지 않는다

| 대상 | 판정 |
|---|---|
| **Rebate Golden** | 🔴 `NOT_APPLICABLE` — 구현 0 |
| **Preservation Golden** | 🟠 `PARTIAL` — **e2e 3계층 REAL**(GATE 5) |
| **보존 목록 무결성** | 🔴 `FAILED` — 팬텀 1 + stale 1 |

**한 마디로 덮으면 두 가지를 놓친다:**
① **Preservation Golden은 이미 부분 REAL**이다 — `npm run e2e`(266차)가 GATE 5로 돌고 있고,
이를 "없음"으로 보고하면 **266차·281차 자산 부정 = 무후퇴 위반**.
② **보존 목록 자체가 검증에 실패**했고, **그것이 골든 작성보다 먼저**다.

### D-2. 🔴 Golden 은 두 종류 — 하나는 불가능, 하나는 **지금 가능**

**Rebate Golden은 1-7과 같은 이유로 불가능하다**(테스트할 코드가 없다).
**그러나 Preservation Golden은 오늘 만들 수 있다 — 기존 시스템은 지금 운영에서 돌고 있다.**

### D-3. 🔴 GOLDEN-GAP-01 — 보존 대상에 팬텀이 있다

**5-1 회귀 게이트의 "보존 대상(회귀 0 검증 대상)"에 `tools/guard_headerless_getjson.mjs`(275차)가 포함**돼 있다.
**그 가드의 호출처는 0이다.**

> **실행되지 않는 것은 회귀할 수 없다.** 회귀 0 검증은 **공허하게 참**이고,
> 골든 케이스를 붙여도 **항상 통과**한다 — **초록인데 아무것도 지키지 않는다.**
>
> **5-8의 고아 가드 발견이 여기서 2차 피해를 드러낸다:** 가드가 안 도는 것에 더해
> **회귀 게이트가 그것을 보호막으로 계산에 넣고 있었다.**

**제거가 아니라 배선(MR-1-7-05)이 선행**돼야 한다. **목록에서 지우면 275차 의도가 사라진다**(무후퇴 위반).

### D-4. 🔴 GOLDEN-GAP-02 — "실측" 351은 실측이 아니라 **코드 주석**

5-1 회귀 게이트의 **"★회귀 고위험 지점(실측)"** = `requirePro` 호출부 **351**.

**출처 추적:**
```
backend/src/Handlers/UserAuth.php:329
  // ... 기존 351개 호출부의 실효 동작(무료/데모 차단·유료 통과)을 보존한다.   ← 286차 주석
```

**289차 실측(방법 명시): A=498 · B=467 · C=458** (5-6 측정 455 · 방법 미기록).
**어느 방법으로도 351보다 100 이상 많다** → **회귀 범위를 351로 잡으면 ≈30% 누락.**

**5-6이 이미 잡은 오류의 재발이다**(주석 351 → 실측 455). **정정이 5-1 §53 문서에 전파되지 않았다.**

### D-5. 🔴 GOLDEN-GAP-03 — 방법 없는 수치는 골든이 아니다

**정직한 grep 3개가 498·467·458을 낸다. 셋 다 옳다** — 세는 대상이 다를 뿐이다.
**method를 안 적으면 455와 498은 모순으로 보이지만 둘 다 옳을 수 있다.**

> **골든의 정의는 "재현 가능"이다.** 재현 불가 기준선은 **골든이 아니라 스냅샷**이다.
> 다음 세션이 같은 숫자를 못 내면 **"회귀"인지 "다르게 셈"인지 구분 불가** →
> **골든이 경보가 아니라 소음이 되고, 소음은 결국 꺼진다.**
>
> **351이 3개 차수를 살아남은 방식이 정확히 이것이다 — 명령이 없어 재현도 반증도 불가능했다.**

**1-6 E-03의 `method` 필수 규정이 여기서 실증됐다.**

### D-6. `VACUOUS` verdict 신설

팬텀 대상 골든은 `PASS`가 아니라 **`VACUOUS`**.
**"통과"로 기록하면 팬텀이 영원히 안 보이고 골든 커버리지 %가 거짓이 된다.**

### D-7. `CorrectionPropagation` — 수치를 문서에 복사하지 않는다

**5-6은 351이 틀렸음을 발견했으나 5-1 §53 문서는 여전히 351이다.**
**틀린 값이 여러 문서에 복사돼 있으면 한 곳을 고쳐도 나머지가 살아있고, 다음 사람은 정본을 모른다.**
→ **수치는 `MeasurementMethod` 참조로 두고 값을 복사하지 않는다.**

### D-8. 순서 — 목록을 먼저 고친다

**팬텀 제거 → stale 정정 → method 도입 → 골든 작성 → Rebate 골든.**
**1·2를 건너뛰고 4로 가면 골든이 팬텀과 stale 값을 정본으로 굳힌다.**
**틀린 목록으로 만든 골든은 틀린 것을 고정한다.**

### D-9. 러너 신설 금지 (Golden Rule)

`e2e/scenarios.mjs` 확장 · `baseline.json` sacred_sha 패턴 확장 ·
`PhantomTargetDetector` = **5-8 `GuardWiringRule` 동일 정본** · `GoldenCertificationRun` = **5-8 동일 정본**.
**재정의 금지.**

### D-10. 351을 고치지 않았다

**문서 수정도 본 세션 범위 밖**이다. **코드변경 0은 문서 정정에도 적용**했다 —
§53 문서는 **1-4/5-1 블록의 산출물**이고, 남의 블록 산출물을 측정 블록이 말없이 고치면
**측정과 수정의 경계가 무너진다.** **발견을 보고하고 승인 세션에 인계한다.**

## 1-8이 인증하지 못하는 것

1. **기대값의 타당성** — **설계가 틀리면 틀린 대로 PASS**.
2. **보존 목록의 완전성** — **1-4/5-1이 열거한 것이 전부라는 보장은 없다.** 누락 검사 **미수행**.
3. **데모/운영 차이** — GATE 5는 **데모 대상**(`deploy.yml:64`). **운영에서만 나는 회귀는 못 잡는다.**
4. **1-8 자신** — **위임도 스펙도 없다.** 1-8의 요구 목록은 저장소에 없다(COV-GAP-01 소급).
