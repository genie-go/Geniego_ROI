# ADR — DSAR Canonical Monetary Reward Entity Model (EPIC 06-A Part 3-3-3-3-3-3-3-3-4-5-1-2)

- **일자**: 289차 (2026-07-16)
- **상태**: Accepted (Canonical Monetary Reward Entity Model 계약 명세 확정. 비파괴 — 코드변경 0). 실 Aggregate/Money VO/Unified Ledger/Participant·Beneficiary/Accrual/Settlement·Payout(outbound)/Clawback/Liability/Accounting/CI 가드 구현은 **현금성 보상 기능(outbound) 도입 시** 후속 승인 세션. **모든 현금성 보상을 단일 상태로 축약·Customer를 Beneficiary/Payout Recipient로 자동 동일시·Commission Recipient(Affiliate/Creator/Seller/Partner)를 Customer로 통합·Accrual을 Payout으로·Settlement을 Payout으로·Wallet Credit을 Bank Payout으로·Face Value를 Paid Value로 혼용·Payment Transaction을 Monetary Reward Transaction으로 혼용·Ledger Entry 수정/삭제·Idempotency 없는 Transaction·Reversal/Clawback을 원 거래 없이 기록·Currency 없는 Amount·FX Version 미보존·금융계좌 원문 저장·inbound 정산(pg/kr)을 outbound reward로·광고 지출을 reward로 매핑·기존 Money/Currency/Ledger 모델 중복 생성·현금성 보상 엔진 부재를 NO_DATA/오탐 처리 금지.**
- **근거(실측)**: [`../segmentation/CANONICAL_DSAR_MONETARY_REWARD_ENTITY_MODEL.md`](../segmentation/CANONICAL_DSAR_MONETARY_REWARD_ENTITY_MODEL.md) · [`LEDGER_GOVERNANCE`](../segmentation/CANONICAL_DSAR_MONETARY_REWARD_LEDGER_GOVERNANCE.md) · **class Money/Unified Ledger/Participant/Beneficiary/Accrual/outbound Payout/Clawback/Liability/Balance 부재(grep 0)** · `pg_settlement`(gross/fee/net·currency·**UNIQUE(tenant,provider,txn_id)** 멱등·inbound) · `kr_settlement_line`(platform_fee/ad_fee/shipping_fee/return_fee·**net_payout**·currency KRW·이중차감 방지 vat/coupon/point 제외·inbound) · `ad_spend_ledger`(광고 지출·billing_key) · `Connectors::fxToKrw`(KRW 단방향·rate version 미보존) · Referral(referrer/referee 분리·비현금 coupon) · OrderHub 취소 역분개·CRM LTV 역분개(Part 268/263)·Pnl net_profit 분류 · Part 4-5-1-1 Provider Inventory · EPIC05 Identity.

## 결정 (핵심)

1. **★실측 정직·outbound 엔진 부재·inbound 정산과 분리**: **현금성 보상(outbound: 테넌트가 고객/파트너/추천인/제휴사/셀러/크리에이터에게 지급)을 발생·적립·정산·지급하는 Canonical 엔진은 부재** — Aggregate/Program/Account/Participant/Beneficiary/Definition/Accrual/Unified Ledger/Balance/Snapshot/Reservation/outbound Settlement·Payout/Reversal/Clawback/Liability/Accounting/Funding Allocation/Money VO 전부 NOT_APPLICABLE(grep 0). 실체=**inbound 정산**(`pg_settlement`·`kr_settlement_line` — 테넌트가 PG/마켓에서 **받는** 돈)·광고 지출(`ad_spend_ledger`)·FX(fxToKrw)·Idempotency(UNIQUE)·역할분리(Referral)·역분개(OrderHub/CRM)·P&L 분류(Pnl). **★핵심 구분: pg/kr 정산은 inbound(수취)·outbound 보상 아님 → Ledger/Currency/FX/Idempotency/Settlement 패턴의 참조 정본이나 도메인 분리(KEEP_SEPARATE)**. 지어내기·NO_DATA/오탐 금지·본 Entity Model=멀티테넌트 고객용 미래 현금성 보상 공통 전방호환 계약.

2. **기존 Money/Currency/Ledger 재사용·중복 금지(§4.10·§50)**: **class Money VO 없음**(DECIMAL+currency 컬럼)→**단일 Canonical Money VO 신설**(중복 금지). FX=fxToKrw 단일(확장·rate source/version/timestamp/gain·loss 추가)·Currency Enum 난립 없음·Ledger=도메인별 개별 테이블(pg/kr/ad_spend)→**Unified Monetary Reward Ledger로 통합 계획**(즉시 삭제 금지·Migration/Compatibility). Reward 유형별 독립 Candidate Store/Ledger 금지.

3. **Participant≠Beneficiary·Customer≠Commission Recipient(§4.2·§4.8·§5)**: Reward 발생 Customer ≠ 실제 지급 Beneficiary·Payout Recipient. Buyer/Referrer/Referee/Affiliate/Influencer/Creator/Seller/Partner/Employee/Organization/Payout Recipient 역할 분리. Affiliate/Creator/Seller/Partner=별도 Participant Identity·Account Binding(Customer 자동 통합 금지). **현행 실 사례=Referral referrer(수령)≠referee(발생) 분리**. 범용 Participant/Beneficiary=신설.

4. **Lifecycle 상태 분리·Accrual≠Payout(§4.1·§4.4·§7)**: Calculated/Estimated/Accrued/Pending/Approved/Available/Reserved/Redeemed/Payable/Settled/Paid/Reversed/Clawed Back/Expired/Written Off/Failed 별도. Accrual 후 Return Window/Fraud/Approval Hold·Min Threshold·Recipient Verification·Tax Review·Settlement Delay·Payout Failure로 미지급 가능. 현재 지급상태만 저장 금지·전 Lifecycle 보존. **RETURN_WINDOW_HOLD=kr_settlement_line return_fee·환불 역분개(Part 263) 인접 근거**.

5. **★Append-only Ledger·Idempotency·Transaction≠Payment(§4.5·§4.6·§19·§20)**: 모든 Monetary Ledger Append-only·Sequence·Idempotency Key·Effective Time·Source Event·**Balance After** 기록. 기존 Entry 수정/삭제 금지→Reversal/Correction/Adjustment/Clawback Entry 추가(Point Ledger Part 4-3 정합). **Monetary Reward Transaction≠Payment Transaction**(연결하되 Entity/Ledger 분리). Source Balance vs Ledger-calculated Balance 비교(Snapshot/Checkpoint). **현행 pg_settlement UNIQUE·append/upsert=Idempotency/Append 패턴 참조 정본**.

6. **Settlement≠Payout·Wallet Credit≠Bank Payout·금융계좌 원문 금지(§4.7·§26·§28)**: Settlement(Gross/Adjustment/Reversal/Clawback/Fee/Withholding/Net Payable)와 실제 Payout(Failed/Returned/Retried/Reversed 상태) 분리. Internal Wallet Credit/Store Credit/Invoice Credit ≠ Bank/Provider Payout/Cash Equivalent. **Payout Destination=Masked Reference·Verification 상태만(금융계좌 원문 저장 금지)**. Reversal/Clawback=반드시 Original Reward/Accrual/Transaction/Settlement/Payout 연결(Runtime Guard). **현행 pg_settlement(gross/fee/net)·kr_settlement_line(net_payout·fee/tax 분리)=Settlement 필드 참조 정본(단 inbound)·outbound Payout=신설**.

7. **Currency·FX Version 강제(§4.9·§37·§38)**: 모든 Monetary Amount에 Amount·Currency·Precision·Rounding Rule·Value Type·Effective Time 강제·Currency 없는 금액 금지. Face/Estimated/Accrued/Approved/Payable/Paid/Funded/Liability/Accounting/Display Value 분리. Historical Reward=당시 FX Rate Version 보존. **현행 fxToKrw=변환 실 구현(KRW 단방향)·rate source/version/timestamp/gain·loss 미보존=GAP**→Canonical FX Conversion 신설.

8. **정직·무후퇴·Coverage/Gap·Funding/Liability/Accounting**: outbound 현금성 보상 엔진 부재=결함 아니라 **`PROVIDER_LIMITATION`/NOT_APPLICABLE 명시**(NO_DATA/오탐 금지). Funding Allocation(Platform/Merchant/Partner/Seller/Shared-funded)·Liability(accrued/payable/breakage)·Accounting Reference(journal/classification)=부재·신설(Rollup net_profit·Pnl 분류 인접). Candidate/Evidence=전방호환. Cross-Tenant/Wrong Beneficiary/Unverified Recipient/Currency 없음/FX 없음/Duplicate/Invalid Reversal·Clawback=Runtime Guard·Critical Gap 시 Access Review 차단. pg_settlement/kr_settlement_line/ad_spend_ledger/fxToKrw·`/v427/pg/*`·Rollup·BillingMethod·Pnl API 보존(회귀 0). 기능후퇴 0.
