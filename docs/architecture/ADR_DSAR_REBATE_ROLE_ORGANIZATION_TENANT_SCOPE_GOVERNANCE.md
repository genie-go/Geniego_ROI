## 결정 (핵심)

1. **★1차 grep REAL 4건 중 3건이 오탐이었다(정직)**: **Workspace·Store·Country Registry 는 부재**다. `WorkspaceState` 는 **테넌트 KV 저장소**(UI 상태)이고 주석이 "테넌트 격리 전용 tenant_kv 신설"이라 명시하며, `influencer_store` 는 **인플루언서 스토어**, `country` 는 **컬럼**일 뿐이다. **이름이 같다고 조직 Registry 가 아니다** — 오탐을 REAL 로 표기 금지(FP 레지스트리 규약·`feedback_competitive_gap_verify` 정합).

2. **조직 축 실측 지형**: **REAL = Tenant(auth_tenant 위조불가 주입 + `tenant_business_profile`) · Team(`team` 테이블 + team_role + team_channel_mapping) · Brand(catalog_brand) · Agency↔Client(agency_client_link)**. **부재 = Organization · Workspace · Department · Store · Country/Region Registry · Legal Entity · Environment 권한 축**. **현행 조직 트리는 Tenant → Team → User 2단**(team_channel_mapping 의 tenant_id+team 복합키가 근거).

3. **★Tenant Registry 는 신설이 아니라 확장**: `tenant_business_profile` 이 이미 company_name·**biz_reg_no**·industry·country·brand_name 을 보유(272차) → **별도 Tenant Registry 신설 금지·이 테이블 확장**(Golden Rule=Extend). **단 `biz_reg_no` 가 있어도 Legal Entity 가 아니다** — 법인 회계/책임 주체 ≠ 테넌트 프로필(1-3 §Accounting 정합).

4. **★Scope 2계층 분리(자율 설계의 핵심 판단)**: 현행 `DATA_SCOPES` 9종은 **조직 축**(company·team·partner)과 **업무 데이터 축**(campaign·product·channel·warehouse·own)이 **혼재**하여 5-1 Canonical Scope Dimension 24 와 **차원이 다르다**. 강제 병합 시 의미 손상 → **①조직 Scope(주체 범위) ②데이터 Scope(Resource 범위) 2계층으로 분리**하고 **기존 9종의 의미·값은 변경 금지**(확장만). `own`(본인 데이터만)은 **최협 Scope** 로 특별 취급.

5. **Scope Hierarchy·Inheritance·Resolution**: 목표 트리=Legal Entity→Organization→Tenant→Workspace→Department→Team→User(**현행은 Tenant→Team→User 2단**). **Inheritance 기본 = EXPLICIT_ONLY**(부모 Scope 자동 부여 금지·Least Privilege §4.3). Resolution 은 **Subject Scope ∩ Resource Scope**(비면 DENY·5-1 §32 ⑨⑩) + **결정 경로 기록 필수**(Vol4 Explainable). Conflict 기본 = **RESTRICTIVE_WINS**(5-1 §4.2 정합).

6. **Agency↔Client 가 Tenant Isolation 의 유일한 승인 예외**: `agency_client_link` + **매 요청 approved 재검증 fail-closed** + agency 토큰의 **서버바인딩 tenant 주입 + 최소권한 role**(write 없으면 viewer·index.php:97-100)이 **Cross-Organization 접근의 실 정본**. **★중복 Cross-Tenant 경로 신설 금지** · 그 외 Cross-Tenant = Critical Gap(5-1 §43).

7. **★Role Registry 자체가 없다**: Role 이 **3계통에 enum/상수/컬럼값으로 산재** — ①team_role(**고정 3종** owner/manager/member) ②api_key role(viewer/connector/analyst/admin + scopes) ③admin master/sub. **Custom Role · Role Registry 테이블 · Assignment 이력/승인/만료 · Inheritance 전부 부재**. → **Canonical Role Registry 로 통합하되 `legacy_system` + `legacy_value` 로 매핑 보존**하고 **기존 값·의미·판정 코드를 바꾸지 않는다**(회귀 0). **★Role 을 4번째로 늘리지 말 것**(5-1 §51).

8. **★대량 회귀 위험(실측·통합 시 최우선 고려)**: `requirePro` **호출부 351개**(**286차 rank 맵 붕괴 실사례** — starter=growth=pro=1 로 붕괴돼 requirePro 가 사실상 'starter+' 였다) · `authedTenant` **64 핸들러** · `requireMasterAdmin2` **5 핸들러** · index.php **bypass 목록**(275차 헤더리스 getJson 401 회귀 2차 재발). → **의미 변경 금지 · 매핑만 · Legacy Equivalence(5-8) 필수**.

9. **Custom Role 신설 + 상한 강제**: 멀티테넌트 고객사 자체 Role 요구의 핵심이나, **`max_privilege_level` 상한 필수**(Custom Role 로 플랫폼 관리자 권한 획득 = **권한상승**). 상속으로 sensitive/production Role 자동 획득 차단.

10. **Role Assignment Governance 신설**: **assigned_by · assignment_reason · approval_reference · valid_from/to · Evidence 없는 부여 금지** · **자기 부여 금지**(Maker-Checker 상세=5-4) · **sensitive/production Role 은 Time-bound 필수**(5-5 JIT) · Assignment History **Append-only** · **Expired/Revoked 로 접근 지속 = Critical**(5-1 §43).

11. **★Privilege Escalation Guard(8종) — 192차 정본 승격**: **SCOPE_GATE_BYPASS** 의 실 사례가 **192차 보안 P0** — `/v421/keys` 게이트가 **`/api` 별칭으로 우회**되어 일반 `write:*` 키가 **admin 키를 발급**할 수 있었다(index.php:562-567). **★영구 규칙: 신규 권한 게이트 작성 시 `/api` 별칭 변형 동시 매칭 필수**. 그 외: SELF_ASSIGNMENT · CUSTOM_ROLE_OVERREACH · INHERITANCE_ESCALATION · **FAIL_OPEN_ATTRIBUTE** · CROSS_TENANT_ROLE · ENVIRONMENT_ESCALATION · APPROVAL_BYPASS.

12. **★`team_role` fail-open 판정(5-1 §0-3 위임 수령·자율 판단·미확정)**: 실측 = "fail-open: team_role 미설정=레거시 단독회원=**owner**"(AdminMenu.php:52-54). **판정: 의도적 설계로 보인다** — 레거시 단독회원(팀 개념 이전 가입자)은 team_role 컬럼이 없어 owner 로 해석하지 않으면 **자기 테넌트에서 잠긴다**(가용성 보호). **그러나 5-1 §4.1 Deny-by-default · §24 null_behavior=DENY 와 상충**. **권장 해소(자율)**: fail-open 제거 대신 **부재 시 owner 로 명시적 backfill(1회 마이그레이션) → 이후 null=DENY 전환** = 가용성 보존 + 계약 정합. **★본 세션 미수정**(비파괴) · **FP 규약상 PM 코드 재증명 전 P0 단정 금지** · **실 전환은 별도 승인 세션**(회귀 위험: backfill 선행 없이 전환 시 **레거시 단독회원 전원 잠김**).

13. **sub-admin 스코프 강제는 REAL(재사용)**: 286차 — `requireMasterAdmin2`=메뉴트리/부여/요금/DB/쿠폰쓰기 **master 전용** · `requireSubAdminMenu`=growth/site-intro/legal-docs/plan-pricing **부여경로만** · **키=ADMIN_MENU `it.to` 정합 오차단 0** 검증 완료. → Canonical Role 통합 시 **이 스코프 경계 보존**.

14. **정직·무후퇴**: **코드변경 0**(문서 전용·회귀 0). tenant_business_profile/team/team_channel_mapping/acl_permission/api_key/agency_client_link/catalog_brand/requireMasterAdmin2/requireSubAdminMenu/PlanPolicy/authedTenant 보존. **스펙 §요구가 없으므로 "완료 조건" 판정 불가** — 본 산출은 **자율 설계 계약 명세**이며 **실 코드·테이블·Lint·Guard 0건**.
