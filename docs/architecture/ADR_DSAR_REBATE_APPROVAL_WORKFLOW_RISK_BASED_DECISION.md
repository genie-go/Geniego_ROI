## 결정 (핵심)

1. **★승인 워크플로는 부재가 아니다 — 정족수까지 이미 실재(정직)**: **`required_approvals INT NOT NULL DEFAULT 2`**(Db.php:634) + **누적 판정**(`count($approvals) >= required_approvals ? approved : pending`·Mapping.php:214) = **Multi-Level Approval 의 실 정본**. **★본 세션 1차 grep 에서 "Multi-level/quorum 부재"로 판정했으나 오판**이었다 — `quorum`/`approval_level` 만 검색해 **`required_approvals` 를 놓쳤다**. 정밀 확인으로 정정(5-2 의 오탐 3건 정정과 동일 클래스: **키워드 1차 grep 을 결론으로 삼지 말 것**).

2. **승인 요청 2계통 통합(CONSOLIDATION_REQUIRED)**: **`action_request`**(policy_id·action_json·approvals_json·**requested_by 없음**) + **`mapping_change_request`**(requested_by·approvals_json·**required_approvals**·**policy_id 없음**) → **Canonical Approval Request 가 상위 집합**. **기존 테이블·필드·판정 코드 변경 금지**(회귀 0 · 5-2 Role 통합과 동일 규율).

3. **★Quorum ≠ Level(자율 판단)**: 현행 `required_approvals` 는 **인원 수(count)**일 뿐 **단계(level)·역할(role)이 아니다**. "2명 승인"과 "Finance 1명 + Legal 1명 승인"은 **다른 요구**다. → **Approval Level(level_order·required_role·required_scope·required_clearance·sequential/parallel·escalation_after) 신설** · 현행 count 로직은 **Level 내 required_count 로 편입**(의미 변경 금지).

4. **★Quorum Policy 에 3종 신설 필수(§7 관찰 기반)**: **`distinct_approver_required`**(동일인 중복 승인 차단·**현행 부재**) · **`exclude_requester`**(자기 승인 차단·**현행 부재**·5-4) · **`required_roles`**(**누가** 승인해야 하는지·현행은 count 만).

5. **★★인접 관찰(미확정·본 파트 최대 발견) — `Mapping::approve()` 중복/자기 승인 방지 부재**: `approve()` **전체**(Mapping.php:196-216) 확인 결과 —
   - **중복 승인 방지 없음**: `$approvals[] = ["user"=>$actor, "ts"=>gmdate('c')];`(:212) 에 **동일 `$actor` 검사 없음** → **한 사용자가 approve 를 2회 호출하면 `count($approvals)=2` 로 `required_approvals` DEFAULT 2 를 혼자 충족 → `approved`**(:214).
   - **자기 승인 방지 없음**: `requested_by`(:167 기록)와 `$actor` 를 **비교하지 않음** → **요청자 본인이 승인 가능**.
   - **정직 표기**: **FP 레지스트리 규약상 PM 코드 재증명 전 P0 단정 금지**. 실 영향 판정에 필요 = ①approve 엔드포인트 **호출 권한**(5-1 §PEP 100+ 분산 상태) ②`mapping_change_request` **실 운영 사용 여부** ③`self::actor()` 해석. **본 세션 비파괴·미수정** → **MIGRATION_REQUIRED · 5-4(Maker-Checker/SoD) 판정 대상**. **정족수 로직 자체는 정본이므로 재사용하되 5-4 에서 dedup + self-approval 가드 추가**.

6. **★"승인됨" ≠ "집행됨"(287차 정본 승격)**: 승인 후 상태는 **실 집행 결과로 정직 기록** — `executed`=실집행성공 / `failed`=실패 / `approved_manual`=자동집행 불가(Alerting.php:610-611). **287차 가짜집행**("status='executed' 로만 바꾸고 어떤 외부 API 도 호출하지 않아 fake-looks-real")이 **이 계약의 실 사고 근거**. State 10 에 **APPROVED 와 EXECUTED/FAILED 를 분리**.

7. **승인 만료·철회 신설**: **현행 부재** — 승인 요청이 **무기한 pending** 가능. `expires_at` + EXPIRED + 재요청 경로 · **만료된 승인으로 집행 금지**(5-1 §18 "승인 만료 후 발효 금지" 계승). 철회는 요청자만 · **이미 APPROVED 면 Rollback 경로**(1-4 §37).

8. **★Risk-based 승인은 부재 — `risk_score` 는 오탐(정직)**: `risk_score` 히트는 **CustomerAI 의 이탈 예측(churn_prob)** 이며 **승인 위험과 무관**. **동명이의를 REAL 근거로 삼으면 오탐**(FP 규약·5-2 오탐 3건과 동일 클래스). → Risk Model(14 Factor)·Level(6·**UNKNOWN=CRITICAL fail-closed**)·Score Snapshot(Decision 시점 고정·모델 버전 기록) **신설**. Risk Signal 원천 = **AnomalyDetection · Alerting condition_tree**(재사용).

9. **★Threshold 정본 재사용**: **`enforceBudgetCaps` 97%**(284차 예산 하드상한) · **MTD budget cap** · **`charging` 선점** = **금액 임계 + 초과 차단**의 실 패턴. Threshold Type(8) · **초과 시 동작 명시 필수**(BLOCK/REQUIRE_APPROVAL/REQUIRE_STEP_UP/MANUAL_REVIEW·암묵적 허용 금지) · **금액은 Decimal 또는 정수 minor unit**(Float 금지·선행설계 R3 §43) · **★임계 집계에 in-flight 포함 필수**(R3 §53 "빠지면 이중청구의 원인이었다" 클래스). 5-1 §34 정합: Funding 변경 금액 **Threshold 초과 시 REQUIRE_APPROVAL**.

10. **★자동 승인 ≠ 자동 집행(본 파트 최우선 계약)**: 자동 승인은 **승인 단계를 생략**할 뿐 **집행을 보장하지 않는다**. **"자동 승인됨"을 "집행 완료"로 표기 = 287/288차 fake-looks-real 클래스 재발**. **금지 조건(강행)**: Production 고위험 · 금융/Payout/Credential · Cross-Tenant/Legal Entity 교차 · anomaly_signal 존재 · **Trust READY 미달**(헌법 Vol3) · Stale attribute(5-1 §20) · risk UNKNOWN · distinct_approver/exclude_requester 요구 건 → **전부 REQUIRE_APPROVAL/MANUAL_REVIEW**. **헌법 Vol5 Safety Rule 정합**(신뢰도/권한/동기화/통계신뢰 부족 시 자동집행 금지→경고). **kill switch 필수**.

11. **Risk Escalation·Step-up**: risk_level 상승 → **required_approvals 증가 + required_roles 추가 + Level 추가**(현행은 **고정 DEFAULT 2** · 위험 연동 부재) · SLA 초과 시 상위 이관. Step-up = **MFA 재인증**(현행 `mfa_policy` REAL 재사용) · **Production Credential 사용은 항상 Step-up**(5-1 §34).
