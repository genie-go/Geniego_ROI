## 결정 (핵심)

1. **★실측 정직·Rebate Lifecycle/Version 엔진 부재·인접 재사용**: **Rebate Program Lifecycle/Versioning/Migration 엔진은 부재(NOT_APPLICABLE·grep 0)**. 실 인접=**Snapshot/Rollback(menu_defaults version+snapshot_data+baseline 복원지점)**·**Supersession(catalog_writeback_job 상태 마감·결과 판정 제외)**·Effective-dated Version(kr_fee_rule)·Validity/Expiry(free_coupons valid_until·NULL=무기한)·**Transition Enforcement(auto_campaign kill-switch)**·**Migration Ledger/Rollback/Dry-run(schema_migrations·`-- @rollback`·`--dry-run`)**·Idempotent 진화(ensureTables 73 핸들러)·Version Gate(baseline.json). **기존 menu_defaults/migrate.php/ensureTables/kr_fee_rule 재사용(중복 금지·§40)**. 지어내기·NO_DATA/오탐 금지·본 Lifecycle=멀티테넌트 고객용 미래 rebate 전방호환 계약.

2. **★State≠Version·State≠Provider 문자열(§6·§4.1·§4.2)**: 상태 전이가 자동으로 Version을 만들지 않고 역도 성립(PAUSE=Version 불변·요율 변경=State 불변 Version 증가). State는 Canonical Enum(18)·Provider 문자열 그대로 저장 금지(Mapping+Confidence). Lifecycle State(18)·Transition(12)·Trigger(14)·Guard(13).

3. **★Internal State≠Provider Actual State(§4.3·현행 정본 승격)**: **auto_campaign kill-switch(AutoCampaign.php:602-609)가 플랫폼 push 실패 시 DB 상태를 바꾸지 않고 502 반환**(표기 'paused'인데 플랫폼 계속 집행=광고비 누수 방지·233차 P1)을 **Rebate 전이 계약으로 승격** — Provider/계약 상대방 반영 실패 시 내부 State 전이 금지(발산 금지·Reconciliation 필수).

4. **Approval≠Effectiveness·as-of 조회(§11·§12·§4.5)**: 승인=권한 확보·발효=Effectiveness Window 진입(APPROVED≠ACTIVE·Cashback 4-5-2-3 "Approval≠Availability" 계승)·승인 만료 후 발효 금지. **임의 시점 t의 유효 Version=`effective_from <= t AND (effective_to IS NULL OR t < effective_to)` as-of 조회 계약 강제**(kr_fee_rule effective_from·free_coupons NULL=무기한 규약 계승).

5. **Sunset≠Termination≠Expiry≠Retirement≠Archive≠Delete(§13~§17·§4.6·§4.13)**: Suspension(가역·9 Type)·Resumption(소급=명시 정책만)·Sunset(예고·closed_to_new·grace·successor)·Expiry(자동 만료)·Termination(11 Type·in-flight 의무)·Retirement(참조 전용)·Archive(콜드·법정 보존). **★Program/Version/전이/Evidence 물리 삭제 금지(Append-only·catalog :1187 "미처리 잡 삭제" 회귀 교훈 계승)**. Provider API sunset(AdAdapters v16→v17)≠Rebate Program sunset(혼동 금지).

6. **Grandfathering·In-flight Protection(§18·§19·§4.8)**: UNFAVORABLE/BREAKING 변경=Grandfathering(7 Mode)+통지 필수·기존 참여자 불리 신버전 자동 소급 금지. **Version 전환/Suspension/Sunset/Termination 시 진행 중 Accrual/Claim/Settlement/Payout은 "발생 시점 Version"으로 완결(pinning)**·Terminal 전이가 in-flight 무단 폐기 금지.

7. **Version 무결성·Amendment≠Correction(§20~§26·§4.7·§4.11)**: Version Append-only(삭제/덮어쓰기 금지·새 Version만)·**Snapshot 없는 Version 발행 금지**(menu_defaults baseline 재사용·282차 "스냅샷 없어 reset 404" 교훈=**스냅샷 없으면 롤백 불가**)·Snapshot 불변·**Diff 없는 Version 발행 금지**(승인 불가)·Change Classification(9)·Supersession=물리 삭제 대신 상태 마감+판정 제외(catalog 정본). **Amendment(장래효·새 Version+새 Window·과거 불변)≠Correction(과거 오류 수정·승인+영향목록+역분개 Append-only+Audit)** — 상호 위장 금지·정정도 원본 Version 덮어쓰기 금지.

8. **Migration=Dry-run+Rollback+멱등(§27~§30·§4.9)**: **migrate.php 정본 재사용** — dry-run 결과(:13) 없이 실행 금지·`-- @rollback`+record 제거(:15)/복원 Snapshot 없는 Migration 금지·`schema_migrations` 기적용 skip(:48)=멱등 원장(원장 미기록 실행 금지)·재실행 안전(ensureTables CREATE IF NOT EXISTS 73 핸들러). Plan Type(9)·Legacy Mapping(이름만 EXACT 금지·미검증 매핑 Accrual 금지).

9. **★Backfill≠Recalculation(§31·§32·§4.10·§4.4)**: **현재 Version으로 과거 거래를 재계산해 기존 Accrual/Claim/Settlement를 덮어쓰는 행위 금지**. 허용=①as-of Version 기반 신규 생성(Backfill·승인+dry-run+rollback) ②승인된 Correction(역분개·Append-only·OrderHub 역분개/4-5-2-5 Reversal 정합). 금지=무승인 일괄 재계산·기존 원장 UPDATE·in-flight 신버전 재계산.
