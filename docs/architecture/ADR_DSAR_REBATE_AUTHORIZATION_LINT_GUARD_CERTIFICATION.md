## 결정

### D-1. 신설 금지 — 기존 GATE 확장 (Golden Rule)

현행 게이트가 **REAL**이다: CI **GATE 1~5**(`deploy.yml:45/48/53/59/64`) ·
security-scan(npm audit·CodeQL) · pre-commit **B1~B4** · `baseline.json`(sacred_sha).

→ **새 파이프라인·새 러너를 만들지 않는다.** GATE 2/3/5에 검사를 **편입**한다.
게이트가 두 벌이면 **정본이 사라진다**(288차 가짜녹색 재발 경로).

### D-2. 🔴 `guard_headerless_getjson.mjs` — 고아 가드 확정

**부재증명:** 파일 실재(`tools/`), **호출처 0**(`.github/` · `.githooks/` · `package.json` 전수 grep).
**대조군:** `guard_channel_writeback.mjs`는 **`.githooks/pre-commit:175` 배선 REAL**.

→ 275차 이래 **단 한 번도 자동 실행되지 않았다.** **인계서의 "REAL" 기록은 파일 실재만 본 서술**이었고
5-8이 정정한다. **배선은 코드변경**이므로 승인 세션 인계(**MR-5-8-01 · 1줄**).

### D-3. pre-commit은 강제가 아니다 — B4의 CI 승격

CI에서 hook 미실행(grep 0) + `core.hooksPath`는 **클론별 로컬 config** + `--no-verify` 우회 명시.
→ **hook 유지**(수초 피드백은 고유 가치) + **B4(secret 차단)는 CI GATE로 복제**, **로직은 단일 공유**.

### D-4. `GuardWiringRule` — 메타 가드

D-2의 교훈은 "가드를 더 만들자"가 아니라 **"가드 배선을 아무도 검사하지 않았다"**이다.
→ `tools/guard_*.mjs` **호출처 0 = BLOCK**. 이 규칙이 해당 유형을 영구 차단한다.

### D-5. Ratchet — BLOCK 즉시 적용 금지

신규 Lint를 즉시 BLOCK하면 레거시 위반으로 파이프라인이 마비되고, **개발자는 Lint를 끈다.**
→ **R0 측정 → R1 baseline 동결 → R2 신규 위반만 BLOCK → R3 감축**.
**R2가 핵심 — 출혈을 먼저 멈춘다.**

### D-6. 🔴 DENY 우선 (Golden Dataset)

**ALLOW만 있는 권한 테스트는 무의미하다** — ALLOW는 기능 테스트가 어차피 커버한다.
**권한의 존재 이유는 DENY이고, DENY는 정상 사용 중 실행되지 않으므로 명시 검증 없이는 영원히 미검증**이다.
→ **DENY 케이스 ≥ ALLOW 케이스**. `expected_reason` 필수(엉뚱한 이유의 DENY = 초록인데 고장).

5-7의 **"ALLOW 샘플링 · DENY 전량"**과 동일 원리다.

### D-7. 라우트 자동 도출 (`render.mjs` 패턴 차용)

권한 매트릭스를 **손으로 나열하면 신규 라우트가 영원히 누락**된다 → 가장 안 지켜진다.
→ 281차 `render.mjs`의 **119 라우트 자동 도출** 패턴을 negative matrix에 적용.

### D-8. C-1~C-4 = 100% (완화 불가)

**권한은 부분 통과가 의미 없다.** 99% 막고 1% 뚫리면 **그 1%로 전부 뚫린다.**
성능·UX 지표와 달리 **비율 완화가 성립하지 않는 도메인**이다.

**단 C-7(Lint Coverage)은 임계 미설정** — 임계를 세우면 **측정 가능한 규칙만 쓰는 왜곡**이 생긴다.
"위임은 업무상 필요가 있을 때만" 같은 규칙은 자동 검사가 **불가능**하다. **측정하되 강제하지 않는다.**

### D-9. 인증은 커밋 SHA에 결속

코드가 바뀌면 인증은 **무효**다. **"작년에 인증받았다"는 증거가 아니다.**
→ GATE 5에서 **매 배포 재실행**.

## 거부한 대안

| 대안 | 거부 사유 |
|---|---|
| 전용 authz-lint CI 워크플로 신설 | 게이트 2벌화 → 정본 소실. GATE 2/3 확장으로 충분 |
| pre-commit 폐기 후 CI 일원화 | 수초 피드백 상실. **이중화가 정답**(hook=편의 · CI=강제) |

| ALLOW 중심 테스트 | 권한 시스템의 실패 모드를 전혀 못 잡음 (D-6) |
| 본 세션에서 배선 1줄 수정 | **코드변경 0 원칙 위반**. 1줄이라도 승인 세션 대상 |

## 5-8이 인증하지 못하는 것 (정직 표기)

1. **규칙 자체의 타당성** — 설계가 틀렸으면 **틀린 대로 100% 통과**한다.
2. **미설계 영역** — 케이스가 없으면 검증도 없다.
3. **GATE 5는 데모 대상**(`deploy.yml:64`) — ABAC 속성이 실데이터 의존이면 **운영과 결과가 다를 수 있다**.
   이를 "인증됨"으로 읽으면 안 된다.
4. **RP-001류 절차 결함** — Lint로 못 막는다. **절차는 절차로 막는다**(REPEAT_PROBLEM_HISTORY 확인 고정).

> **모든 것을 자동화로 막을 수 있다고 주장하지 않는다.**
