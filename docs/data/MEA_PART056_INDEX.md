## ★★핵심 설계 제약 5종 (구현 착수 시 필수)
1. **감사 체인 이원화 금지**(D-3) — AI_AUDIT은 **새 해시체인이 아니라 `SecurityAudit` 위에 AI 이벤트 타입**을 얹는다. 체인 정본은 하나여야 tamper-evidence가 성립([[reference_menu_audit_log_not_tamper_evident]]).
2. **로그 3원화 금지**(D-4) — **053 Gateway 일원화와 동시에** 감사 스키마 통일. Gateway가 단일 통과점이 되면 **모든 AI 호출이 자동 감사**(구조적 해결).
3. **승인 경로 이원화 금지**(D-5) — `action_request` 확장(054 D-5와 동일). ★단 **액션 승인 ≠ 모델 배포 승인**이므로 스코프 분리 필요. 생산자 부재는 287/288차 확정 보류분(재플래그 금지).
4. **AI Trust Score는 실 로그 파생만**(D-6) — `security_audit_log`·`ai_analyses`·`ml_model_metrics`·`ai_usage_quota`·`risk_prediction`. ★**임의 수치 금지 — 지어내면 본 Part(Responsible AI) 자체가 반례**([[feedback_real_value_autoderive]]).
5. **규범 재정의 금지**(D-1) — Policy Manager는 문서 규범을 **기계 집행**할 뿐 새 규범을 만들지 않는다. 코드에서 재정의하면 두 개의 진실=회귀.
※ 부가: 성능 요구(§18 Policy Validation ≤200ms·Compliance ≤500ms)는 **매 요청 경로에 얹히므로** 캐시·비동기 선행 필수(무분별 삽입 시 응답시간 회귀). 거버넌스·감사·Trust API는 **전량 인증 필수 접두 + admin 스코프**(053 D-5 교훈 — 공개 bypass 접두 배치 시 인증 우회)·`/api` 변형 동시 등재.
